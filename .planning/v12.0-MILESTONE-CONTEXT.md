# v12.0 Milestone Context: Quality Enforcement Evolution

**Created:** 2026-04-04
**Source:** Post-v8.0 brainstorm (FEATURE-IDEAS.md), v7.0 ESLint MCP research, v9.0 signal intelligence data
**Status:** Ready for `/gsd:new-milestone`

## The Problem

Quality gates today cover two of three enforcement layers: test verification (test_gate) and diff review (diff_review). The missing layer is static analysis — linting, type checking, and structural quality issues that tests don't catch and diff review can't reliably detect.

Beyond gates, phase transitions are trust-based. When an executor says "done," the system believes it. SUMMARY.md existing is the only structural check. There's no verification that the plan's actual DONE criteria were met by evidence (file exists, test passes, function exported). An agent that declares completion without meeting criteria passes through unchallenged.

Finally, test quality is a point-in-time snapshot. VERIFICATION.md captures test count at phase completion, but there's no trend data. You can't see "v8.0 added 68 tests, v9.0 added 47, rate is declining" without manually reading every verification file across milestones.

## What to Build

### 1. ESLint MCP Gate Integration

Add a PostToolUse hook that fires when .ts/.js files are written. The hook calls the ESLint MCP server (researched and recommended in v7.0 Phase 29) and persists results to gate-executions.jsonl alongside existing gate types.

**What exists already:**
- v7.0 Phase 29 researched 5 MCP servers/tools and recommended ESLint MCP — the evaluation is done
- PostToolUse hook pattern is established (test_gate, diff_review in gate-runner.cjs)
- gate-executions.jsonl schema supports new gate types without modification
- Dashboard Gate Health page aggregates by gate type — new gates appear automatically

**What's needed:**
- New gate function in gate-runner.cjs that calls ESLint MCP on Write events to .ts/.js/.cjs files
- Quality-level gating (fast=skip, standard=warn, strict=block) following established pattern
- ESLint MCP server added as a dependency or documented as a prerequisite
- Tests covering gate firing, result persistence, and quality-level filtering

**Effort:** M

### 2. State Machine Guards for Phase Transitions

Before a phase transitions to "complete," structurally verify that plan DONE criteria were actually satisfied — not just that the executor declared them done.

**How it works:**
- Each plan task has a `<done>` tag defining completion criteria
- At phase verification time (execute-phase step verify_phase_goal), the verifier already checks must_haves
- State machine guards add a pre-transition check: parse each task's `<done>` criteria, verify against the codebase (file exists, function exported, test passes, endpoint responds)
- If any DONE criteria are unverifiable, the transition blocks with a specific report of which criteria failed

**What exists already:**
- Plans already have `<done>` tags per task with specific criteria
- The verifier agent already reads must_haves — guards extend this to task-level DONE criteria
- execute-phase workflow has a verify_phase_goal step that's the natural insertion point

**What's needed:**
- DONE criteria parser that extracts verifiable assertions from `<done>` tags
- Verification engine that checks assertions against codebase (file existence, grep for exports, test execution)
- Integration into verify_phase_goal step as a pre-check before the verifier agent runs
- Fallback for ambiguous DONE criteria that can't be mechanically verified (surface for human check)

**Effort:** M

### 3. Test Coverage Trending

Track test count at each phase completion and surface the trend in `/gsd:progress` and `/gsd:digest`.

**How it works:**
- At plan completion (execute-plan summary step), count tests via `node --test --dry-run` or parse test runner output
- Write test count to phase-benchmarks.jsonl alongside existing metrics (correction count, gate fires, quality level)
- `/gsd:progress` shows a sparkline or delta: "Tests: 1028 (+47 since v9.0)"
- `/gsd:digest` surfaces trend: "v7.0: +0, v8.0: +68, v9.0: +47 — rate declining"

**What exists already:**
- phase-benchmarks.jsonl (v9.0) already stores per-plan execution metrics
- `/gsd:digest` already reads phase-benchmarks.jsonl with N>=5 sample guard
- VERIFICATION.md captures test count but doesn't trend it

**What's needed:**
- Test count extraction at plan completion (parse test runner output or dry-run)
- New field in phase-benchmarks.jsonl entries: `test_count`, `test_delta`
- Trend computation in a reader function (aggregate across milestones)
- Surface in `/gsd:progress` and `/gsd:digest`

**Effort:** S

## Dependencies

- v7.0 ESLint MCP research (complete — Phase 29 evaluation done)
- v8.0 gate enforcement hooks (complete — PostToolUse pattern established)
- v9.0 phase-benchmarks.jsonl (complete — schema extensible for test counts)

## Estimated Scope

3 phases, 3-5 plans. All three features extend existing infrastructure rather than creating new systems.

## Key Decision Points

1. **ESLint MCP as runtime dep or external prerequisite?** If runtime dep, the dashboard/installer needs to ensure it's available. If prerequisite, users install it themselves and the gate degrades gracefully if missing.

2. **State machine guards: block or warn?** Blocking prevents false completions but could stall execution on ambiguous DONE criteria. Warning surfaces the issue without blocking. Recommendation: warn in standard mode, block in strict mode.

3. **Test count source: dry-run or parse output?** `node --test --dry-run` lists tests without running them (fast). Parsing actual test output gives pass/fail breakdown but is slower. Recommendation: dry-run for count, actual output already captured for pass/fail.
