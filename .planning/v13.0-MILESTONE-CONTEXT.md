# v13.0 Milestone Context: Unified Observability & Context Routing

**Created:** 2026-04-04
**Source:** Post-v8.0 brainstorm (FEATURE-IDEAS.md), v9.0 signal intelligence pipeline, v10.0 MCP dashboard
**Status:** Ready for `/gsd:new-milestone`

---

## The Problem

Three independent problems that share a root cause: the system generates rich data but stores it in isolated silos with no unified view or routing intelligence.

### Fragmented event streams
Gates log to gate-executions.jsonl. Corrections log to corrections.jsonl. Sessions log to sessions.jsonl. Benchmarks log to phase-benchmarks.jsonl. Context7 calls log to context7-calls.jsonl. Skill metrics go to skill-metrics.json. Skill scores go to skill-scores.json.

When you want to answer "what happened during plan 42-01?" you have to read 5+ files, correlate by timestamp, and mentally reconstruct the sequence. There's no single chronological view of system behavior. Debugging cross-cutting issues (e.g., "why did this gate fire right after this correction?") requires manual file archaeology.

### Invisible context costs
Skills load at session start but there's no tracking of how much context budget each skill consumes. A skill that costs 4K tokens per session but only fires 2% of the time is invisible waste. v9.0's relevance scoring ranks skills by keyword match, but relevance and cost are different dimensions — a highly relevant skill might also be the most expensive.

### Manual MCP configuration
MCP servers are configured once and every task gets the same servers regardless of what it's doing. A library integration task needs Context7. A filesystem restructuring task doesn't. A database migration needs different tools than a UI component. Today the user configures this manually or just loads everything.

## What to Build

### 1. Formal Workflow Event Journaling

Emit structured events to a single workflow.jsonl as a unified chronological stream.

**Event types:**
- `phase_started`, `phase_completed`
- `plan_started`, `plan_completed`
- `task_started`, `task_completed`
- `gate_fired` (with gate type and outcome)
- `correction_captured` (with category)
- `skill_loaded`, `skill_scored`
- `session_started`, `session_ended`
- `checkpoint_hit`, `checkpoint_resolved`

**Each event:**
```json
{
  "type": "gate_fired",
  "timestamp": "2026-04-04T03:22:15.000Z",
  "phase": "42",
  "plan": "01",
  "task": "3",
  "data": { "gate": "test_gate", "outcome": "passed", "quality_level": "standard" },
  "session_id": "abc123"
}
```

**What exists already:**
- Individual JSONL writers for gates, corrections, sessions, benchmarks (v7.0-v9.0)
- Session IDs in sessions.jsonl
- Phase/plan context available at all event emission points

**What's needed:**
- Central event emitter function: `emitEvent(type, context, data)`
- Integration points in gate-runner.cjs, correction hooks, workflow observe steps, session hooks
- Reader function for querying by time range, phase, plan, event type
- `/gsd:digest` integration: "Plan 42-01 timeline: 6 events across 45 minutes"
- Rotation at configurable entry count (default 10000)

**Key design choice:** workflow.jsonl supplements, not replaces, existing JSONL files. Individual files remain for backward compatibility and focused queries. The journal is the unified view.

**Effort:** M

### 2. Context Budget Optimizer

Track per-skill token cost and surface the top context consumers.

**How it works:**
- At session start (when skills load), measure each skill's SKILL.md + references size in tokens (approximation: chars / 4)
- Write `skill_token_cost` to sessions.jsonl alongside existing `skills_loaded`
- Aggregate across sessions: per-skill average cost, load frequency, cost-per-fire ratio
- Surface in `/gsd:digest`: "Top context consumers: skill-integration (4.2K tokens, loaded 100% of sessions), gsd-workflow (3.8K, loaded 95%)"
- Combine with v9.0 relevance scores: high cost + low relevance = candidate for deferral

**What exists already:**
- v9.0 skill relevance scoring (Jaccard overlap, cold-start floor, dormancy decay)
- sessions.jsonl with skills_loaded arrays
- `/gsd:digest` already surfaces skill-related metrics

**What's needed:**
- Token cost measurement function (read SKILL.md + references, estimate tokens)
- Session-level cost recording in sessions.jsonl
- Aggregation function: cost trends, cost-per-relevance ratio
- `/gsd:digest` section: context budget analysis
- Recommendation engine: "Consider deferring X (high cost, low relevance, low quality score)"

**Effort:** M

### 3. MCP Server Selection Intelligence

Before an executor starts a task, inspect the task type and recommend which MCP servers to load.

**How it works:**
- Define a task-type-to-MCP mapping (similar to CATEGORY_SKILL_MAP from v9.0):
  - Library integration tasks → Context7
  - Database tasks → relevant DB MCP if configured
  - File restructuring → filesystem tools only
  - UI components → browser/preview MCP if available
  - API integration → HTTP/fetch MCP
- At plan execution start, parse task descriptions and file types to classify task type
- Emit recommendation: "This plan touches 3 API endpoints — consider loading HTTP MCP"
- If v10.0 MCP dashboard is running, query available MCP servers via tool call

**What exists already:**
- v10.0 MCP dashboard exposes tool endpoints (list-projects, get-observations, etc.)
- Context7 MCP already configured for library lookups
- Task descriptions available in plan XML (`<action>` tags)
- CATEGORY_SKILL_MAP pattern from v9.0 for classification

**What's needed:**
- Task type classifier function (keyword + file extension heuristics)
- MCP recommendation mapping (task type → MCP servers)
- Integration into execute-plan workflow: emit recommendation before executor spawns
- Optional: auto-configuration if v10.0 MCP tools are available for querying

**Effort:** M

## Dependencies

- v9.0 signal intelligence (complete — skill scoring, session analytics, benchmarks)
- v10.0 MCP dashboard (should be complete — MCP server selection benefits from querying available servers)
- v12.0 quality gates (beneficial but not required — ESLint gate events flow into the journal)

## Estimated Scope

3 phases, 4-6 plans. Event journaling is the heaviest lift (many integration points). Budget optimizer and MCP routing are focused features.

## Key Decision Points

1. **Event journal: synchronous or async writes?** Synchronous is simpler but adds latency to every gate fire and correction capture. Async (buffered flush) is faster but risks event loss on crash. Recommendation: synchronous with a fast-path (append to open file handle, no JSON.parse of existing content).

2. **Token cost: measured or estimated?** True token counting requires a tokenizer. Character-based estimation (chars/4) is fast and close enough for relative comparison. Recommendation: estimation, with a note that absolute numbers are approximate.

3. **MCP routing: advisory or automatic?** Advisory surfaces recommendations and lets the user decide. Automatic configures MCP servers before executor spawn. Recommendation: advisory for v13.0 — automatic adds complexity and risk of misconfiguration.
