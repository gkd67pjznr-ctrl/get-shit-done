<feature>
  <name>Milestone Workspace Scaffold and Conflict Manifest</name>
  <files>get-shit-done/bin/lib/milestone.cjs, tests/milestone.test.cjs</files>
  <behavior>
    cmdMilestoneNewWorkspace(cwd, version, options, raw):
    - Input: cwd (project root), version (e.g., "v2.0"), options (empty object), raw (boolean)
    - Creates directory tree: .planning/milestones/<version>/ with phases/ and research/ subdirs
    - Creates scaffold files (only if not already present):
      - STATE.md: `# Project State — Milestone <version>\n\n**Created:** <date>\n**Status:** Initializing\n`
      - ROADMAP.md: `# Roadmap — Milestone <version>\n\n*Created: <date>*\n`
      - REQUIREMENTS.md: `# Requirements — Milestone <version>\n\n*Created: <date>*\n`
    - Creates conflict.json (only if not already present):
      ```json
      { "version": "<version>", "created_at": "<date>", "status": "active", "files_touched": [] }
      ```
    - Outputs JSON result with version, workspace_dir, created, files, dirs
    - Errors if version is not provided (calls error())
    - Is idempotent: running twice with same version does not overwrite existing scaffold files

    cmdMilestoneUpdateManifest(cwd, version, files, raw):
    - Input: cwd, version (e.g., "v2.0"), files (string array of file paths), raw (boolean)
    - Reads existing conflict.json from .planning/milestones/<version>/conflict.json
    - Merges files into files_touched with deduplication: [...new Set([...existing.files_touched, ...files])]
    - Writes updated conflict.json back
    - Errors if version not provided or conflict.json doesn't exist
    - Outputs JSON result with version, files_touched (updated array), added (count of new files)

    Test cases (input -> expected output):
    - new-workspace v2.0 -> workspace dir exists, all 3 .md files exist, phases/ dir exists, research/ dir exists, conflict.json valid
    - new-workspace v2.0 (run twice) -> second run doesn't overwrite first STATE.md content
    - new-workspace (no version) -> error exit code 1
    - update-manifest v2.0 --files a.js b.js -> conflict.json files_touched = ["a.js", "b.js"]
    - update-manifest v2.0 --files a.js (twice with same file) -> files_touched has only one "a.js"
    - update-manifest v2.0 --files c.js (after a.js,b.js) -> files_touched = ["a.js", "b.js", "c.js"]
  </behavior>
  <implementation>
    Add both functions to milestone.cjs after the existing cmdMilestoneComplete function, before module.exports. Follow the exact same patterns used in cmdMilestoneComplete:
    - fs.mkdirSync with { recursive: true } for directory creation
    - fs.existsSync guard before writeFileSync for idempotency
    - output(result, raw) for JSON output
    - error(msg) for validation failures

    Add both function names to the module.exports block at the bottom of milestone.cjs.

    NOTE: This plan does NOT add CLI routing in gsd-tools.cjs — that is Plan 02. Tests in this plan must call the functions directly (require milestone.cjs and invoke). Do NOT use runGsdTools for these tests since routing doesn't exist yet. Instead, require the module directly:
    ```javascript

    // Call: milestone.cmdMilestoneNewWorkspace(tmpDir, 'v2.0', {}, true);
    ```

    Use `true` for the raw parameter so output() writes plain JSON to stdout. Capture stdout for assertions using the child_process pattern, OR call functions directly and check filesystem state (preferred — simpler, no stdout capture needed).

    For direct function calls: the `output(result, raw)` call writes to process.stdout and does NOT return the result. To test without stdout capture, simply call the function and then assert on filesystem state (fs.existsSync, fs.readFileSync for content).
  </implementation>
</feature>
