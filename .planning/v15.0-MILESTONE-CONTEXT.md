# v15.0 Milestone Context: Autonomous Learning

**Created:** 2026-04-04
**Source:** Post-v8.0 brainstorm (FEATURE-IDEAS.md), v6.0/v8.0 adaptive learning pipeline, v9.0 skill quality metrics
**Status:** Ready for `/gsd:new-milestone`

## The Problem

The adaptive learning pipeline captures corrections, detects patterns, generates suggestions, and refines skills — but every refinement requires a human to type `/gsd:refine-skill accept`. The system can identify that a skill is producing corrections with 0.98 confidence and propose a fix, but it still waits for permission to apply a 3-word change to a SKILL.md file.

This isn't just a friction issue. It's a capability gap. The system has all the data it needs to make safe, evidence-based improvements autonomously — it just lacks the authority and the guardrails to do so correctly.

Three features close this gap at different levels:

1. **Learning loop automation** — Let the system auto-apply high-confidence refinements (the individual skill change level)
2. **Adaptive code review profiles** — Let the system shift review focus per-project based on correction history (the skill behavior level)
3. **Decision audit trail** — Let the system surface when correction patterns contradict recorded decisions (the project strategy level)

Each builds on the previous: auto-apply changes individual skills, adaptive profiles change how skills behave, and decision auditing changes how the project evolves. Together they make the system self-correcting at every layer.

## What to Build

### 1. Learning Loop Automation

For corrections with very high confidence and no controversial flags, auto-apply skill refinements without user confirmation.

**How it works:**
- New config key: `adaptive_learning.auto_apply: false` (opt-in, default off)
- When analyze-patterns.cjs generates a suggestion with confidence > 0.95:
  - Check: suggestion modifies <20% of skill content (existing guardrail)
  - Check: no `controversial` flag on any source correction
  - Check: skill's quality score from skill-metrics.json is "medium" or "low" (don't auto-modify high-performing skills)
  - Check: skill hasn't been auto-modified in the last 7 days (cooldown, existing guardrail)
  - If all checks pass AND `auto_apply: true`: apply the refinement, commit, log to SKILL-HISTORY.md
  - If any check fails: surface as normal suggestion for manual review

**Safety mechanisms (in addition to existing 6 guardrails):**
- Auto-apply log: `.planning/patterns/auto-applied.jsonl` records every auto-applied change with before/after diff, confidence score, source corrections, and reversal instructions
- Reversal command: `/gsd:refine-skill revert <id>` undoes an auto-applied change using the logged diff
- Rate limit: maximum 1 auto-apply per skill per 7-day window (prevents rapid drift)
- Kill switch: setting `auto_apply: false` immediately stops all auto-application; pending suggestions revert to manual

**What exists already:**
- v6.0 bounded learning guardrails (3 corrections min, 7-day cooldown, 20% max change, user confirmation, guardrail logging, 5+ co-activations)
- v8.0 analyze-patterns.cjs auto-trigger via SessionEnd hook
- v8.0 /gsd:refine-skill accept/dismiss flow
- v9.0 skill quality metrics with attribution_confidence
- v9.0 SKILL-HISTORY.md audit trail with unified diffs

**What's needed:**
- Auto-apply decision engine: evaluate all checks, log decision rationale
- Auto-apply executor: apply change, commit, log to SKILL-HISTORY.md and auto-applied.jsonl
- Revert command: parse auto-applied.jsonl, apply reverse diff
- Config key and kill switch behavior
- Tests covering all safety checks, edge cases (conflicting auto-applies, rapid succession)

**Effort:** M

### 2. Adaptive Code Review Profiles

The code-review skill applies uniform review criteria to every project. Adaptive profiles shift the review focus per-project based on that project's correction history.

**How it works:**
- Analyze corrections.jsonl per project: compute category distribution
  - If 40% of corrections are `code.wrong_pattern` → emphasize pattern consistency in reviews
  - If 30% are `code.missing_test` → emphasize test coverage
  - If 25% are `code.security` → emphasize security checks
- Generate a per-project review profile: `.planning/patterns/review-profile.json`
  ```json
  {
    "project": "gsdup",
    "focus_areas": [
      { "category": "code.wrong_pattern", "weight": 0.4, "corrections": 23 },
      { "category": "code.missing_test", "weight": 0.3, "corrections": 17 }
    ],
    "updated": "2026-04-04",
    "sample_size": 58
  }
  ```
- The code-review skill reads review-profile.json and adjusts its review prompts:
  - High-weight categories get explicit check items in the review
  - Low-weight categories get default treatment
- Profile refreshes at session start (or on demand via `/gsd:digest`)

**What exists already:**
- corrections.jsonl with 14-category taxonomy (v6.0)
- CATEGORY_SKILL_MAP in analyze-patterns.cjs (v9.0) — maps categories to skills
- code-review skill (SKILL.md) with review checklist
- Session-start hook that loads skills and could refresh profiles

**What's needed:**
- Profile generator: read corrections.jsonl, compute category weights, write review-profile.json
- Profile reader in code-review skill: load review-profile.json, adjust review focus
- Refresh trigger: session-start hook or `/gsd:digest` step
- Minimum sample guard: don't generate profile with <10 corrections (insufficient data)
- Tests covering profile generation, skill adaptation, edge cases (no corrections, balanced distribution)

**Effort:** M

### 3. Decision Audit Trail

Surface when correction patterns contradict recorded decisions in PROJECT.md.

**How it works:**
- PROJECT.md has a Key Decisions table with decisions, rationales, and outcomes
- corrections.jsonl has correction categories and diagnosis text
- The audit checks: do corrections cluster around topics that relate to recorded decisions?
  - Decision: "Use JSONL not SQLite" + corrections about JSONL parsing → surface tension
  - Decision: "Inline sentinel not separate agent" + corrections about sentinel context overflow → surface tension
  - Decision: "Fast mode = zero behavioral change" + corrections about fast mode behavior → surface tension
- Surface in `/gsd:digest`: "Decision tensions: 2 decisions may need review based on correction patterns"
- Detailed view: which corrections, which decision, what the tension is, how confident the link

**Matching mechanism:**
- Extract keywords from each decision's description and rationale
- Match against correction `diagnosis_text` and `diagnosis_category` using keyword overlap (same Jaccard approach from v9.0 skill scoring)
- Threshold: 3+ corrections matching a decision's keywords triggers a "tension" flag
- Confidence based on match strength and correction count

**What exists already:**
- PROJECT.md Key Decisions table (maintained since v1.0)
- corrections.jsonl with diagnosis_text and diagnosis_category (v6.0)
- Jaccard token overlap scoring (v9.0 skill-scorer.cjs) — reusable for keyword matching
- `/gsd:digest` as the natural surfacing point

**What's needed:**
- Decision parser: extract decisions from PROJECT.md Key Decisions table
- Decision-correction matcher: keyword overlap between decisions and corrections
- Tension scorer: count matching corrections, compute confidence
- `/gsd:digest` section: decision audit with tensions and evidence
- Tests covering matching, scoring, edge cases (no tensions, all tensions, stale decisions)

**Effort:** M

## Estimated Scope

3 phases, 4-6 plans. Auto-apply is the most complex (safety engineering). Adaptive profiles and decision audit are focused features with clear implementation paths.

## Key Decision Points

1. **Auto-apply default: opt-in or opt-out?** Opt-in (default off) is safer and respects the existing user-confirmation philosophy. Opt-out (default on) is more aggressive but violates the v6.0 design principle of "user always in control." Recommendation: opt-in only, with clear documentation of what it does.

2. **Auto-apply scope: per-skill or global?** Per-skill allows enabling auto-apply for stable skills while keeping manual review for volatile ones. Global is simpler. Recommendation: global config key for v15.0, per-skill override as future enhancement.

3. **Review profile: skill modification or context injection?** Modifying the code-review SKILL.md file changes the skill for all projects. Context injection (loading review-profile.json at review time) is per-project. Recommendation: context injection — don't modify shared skills based on per-project data.

4. **Decision audit: PROJECT.md only or also CLAUDE.md?** CLAUDE.md contains coding conventions that are also "decisions." Including both broadens coverage but increases noise. Recommendation: PROJECT.md only for v15.0 — CLAUDE.md conventions are enforced by hooks, not by audit.
