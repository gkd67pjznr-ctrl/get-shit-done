## Summary

Phase 3.2.1 is an urgent inserted phase triggered by a comprehensive manual audit of the `.planning/` directory (documented in `.planning/PLANNING-AUDIT.md`). The audit identified two distinct categories of work: immediate file system cleanup (orphaned files, stale states, misplaced directories) and systemic GSD workflow code fixes (automation gaps that cause planning artifacts to drift out of sync with reality).

The cleanup work is straightforward manual file operations â€” moves, deletes, and text edits. No new abstractions or libraries are required. The flow fixes are surgical Node.js changes to three existing modules (`roadmap.cjs`, `milestone.cjs`, `phase.cjs`) plus minor updates to planning files (`TO-DOS.md`, audit files). The existing test suite (Jest, 14 test files) covers the modules being modified and must remain green.

The critical design insight is that the two categories have different delivery strategies: cleanup tasks are purely documentary/structural with no code changes; flow fixes are code changes that must be tested against the existing test suite. Both categories can be delivered in a single plan given the contained scope.

**Primary recommendation:** Deliver all cleanup and flow fixes in one plan (3.2.1-01), organizing tasks in two waves â€” Wave 1: filesystem cleanup (no dependencies), Wave 2: code fixes with test verification.

### Pattern 1: Plan-Level Checkbox Update (FLAW 1 Fix)

**What:** `cmdRoadmapUpdatePlanProgress` updates the `**Plans:** N/M` text and progress table but never flips individual `- [ ] NN-NN-PLAN.md` lines to `- [x]`.

**When to use:** Called by `phase complete` and `roadmap update-plan-progress` subcommand. Both need the same fix.

**The gap:** In `roadmap.cjs` `cmdRoadmapUpdatePlanProgress` (lines 220-292), when `isComplete` is true, the function updates the progress table row and phase-level checkbox but no code touches the `- [ ] NN-NN-PLAN.md` individual plan lines.

**How to fix:** After the existing plan count update block, add a regex pass that matches `- [ ] {planNum}-PLAN.md` lines within the phase section and flips them to `- [x]`.

**Example approach:**

```javascript
// Source: Derived from existing roadmap.cjs checkbox pattern (line 275-279)
if (isComplete) {
  // Existing: flip phase-level checkbox
  // NEW: flip individual plan-level checkboxes
  // For each plan file found in phaseInfo.plans:
  for (const planFile of phaseInfo.plans) {
    const planBase = path.basename(planFile, '.md'); // e.g. "3.2-01-PLAN"
    const planPattern = new RegExp(
      `(-\\s*\\[)[ ](\\]\\s*${escapeRegex(planBase)}\\.md)`,
      'i'
    );
    roadmapContent = roadmapContent.replace(planPattern, `$1x$2`);
  }
}
```

**Note:** `phaseInfo.plans` is already available in both `cmdRoadmapUpdatePlanProgress` and `cmdPhaseComplete` â€” no new data fetching needed.

### Pattern 2: Milestone Workspace Finalization (FLAW 2/3 Fix)

**What:** `cmdMilestoneComplete` archives ROADMAP.md as-is (frozen at creation) without first updating milestone workspace files to reflect final state.

**When to fix:** `cmdMilestoneComplete` in `milestone.cjs` (line 127-131).

**The gap:** Line 127-131 reads `roadmapPath` (root `.planning/ROADMAP.md`) and writes it to archive. It does not touch the milestone workspace's own ROADMAP (`milestones/v3.0/ROADMAP.md`). The milestone workspace ROADMAP is also never updated during plan execution.

**Scope decision for planning:** The audit identifies two sub-problems:
1. Milestone workspace ROADMAPs are never updated during plan execution (FLAW 2) â€” complex fix
2. `complete-milestone` doesn't finalize workspace before archiving (FLAW 3) â€” simpler fix

**Recommended scope for 16.1:** Fix FLAW 3 only (the simpler archive-time finalization). FLAW 2 (keeping milestone ROADMAPs live during execution) is a larger architectural change that can be deferred to a dedicated phase or handled as part of Phase 17 migration tooling.

### Pattern 3: TO-DOS.md Resolved Item Marking (FLAW 4)

**What:** No workflow marks TO-DOS.md items as resolved when the referenced work is done.

**When to fix:** The simplest fix is a manual edit (not code). The audit notes both INTEGRATION-3 and INTEGRATION-4 are resolved by Phase 3.1 but TO-DOS.md still shows them open. This can be fixed manually as part of the filesystem cleanup wave.

**Code fix option (lower priority):** Could add a `cmdTodosResolve` or inline resolution in `cmdPhaseComplete`. However, given the audit notes this as a systemic gap and TO-DOS.md has no structured format (free-form markdown), automated resolution is low value. Manual edit is sufficient for 16.1.

### Anti-Patterns to Avoid

- **Moving v2.0-phases before confirming v2.0/phases is empty:** `milestones/v2.0/phases/` already exists as an empty dir. Confirm before moving.
- **Deleting v3.0 workspace without understanding impact:** The v3.0 milestone workspace was scaffolded by `new-milestone`. Deleting it won't break anything (the project runs in `legacy` mode and `planningRoot()` resolves to root `.planning/`), but confirm `layout_style === "legacy"` is stable before deleting.
- **Updating v2.0 MILESTONE-AUDIT.md status without confirming the correct file:** There are two copies â€” `milestones/v2.0-MILESTONE-AUDIT.md` (at the milestones root) and `milestones/v2.0/` (does not contain an audit file). Fix the one at the milestones root, then move it.
- **Breaking existing tests:** Any change to `roadmap.cjs`, `milestone.cjs`, or `phase.cjs` must keep all 14 test file suites green. The existing test suite is the regression safety net.

## Common Pitfalls

### Pitfall 1: Moving Files That Are Referenced by Other Files

**What goes wrong:** Moving `v2.0-MILESTONE-AUDIT.md` or `v2.0-REQUIREMENTS.md` to new paths could break any file that references them by path (e.g., links in ROADMAP or MILESTONES).

**Why it happens:** Planning files often contain relative links to other planning files.

**How to avoid:** Grep for references before moving: `grep -r "v2.0-MILESTONE-AUDIT\|v2.0-REQUIREMENTS" .planning/` â€” if no external references found, safe to move.

**Warning signs:** Any `.md` file with a `[text](path)` link to these files.

### Pitfall 2: v2.0/phases/ Is Not Actually Empty

**What goes wrong:** `milestones/v2.0/phases/` might not be empty â€” could have been partially populated. Moving `v2.0-phases/` contents into it could cause collisions.

**Why it happens:** The audit says v2.0/phases/ is empty but this should be verified before moving.

**How to avoid:** `ls .planning/milestones/v2.0/phases/` before any move operation.

**Warning signs:** Any files present in `v2.0/phases/` before the move.

### Pitfall 3: Regex Patterns in roadmap.cjs Breaking on Decimal Phase Numbers

**What goes wrong:** New regex for plan-level checkboxes must handle decimal phase numbers (e.g., `3.2.1-01-PLAN.md`). The `escapeRegex` call handles this, but the pattern must be constructed correctly.

**Why it happens:** Decimal phase numbers contain `.` which is a regex metacharacter.

**How to avoid:** Always use `escapeRegex(planBase)` where `planBase` is the plan filename without extension.

**Warning signs:** Tests for phase `16.1` plans fail while integer phase tests pass.

### Pitfall 4: cmdMilestoneComplete Fix Introduces Dependency on Milestone Workspace Existence

**What goes wrong:** If `complete-milestone` is modified to first update the milestone workspace's ROADMAP, it needs that workspace to exist. For legacy projects (no workspace), this code path must be guarded.

**Why it happens:** `layout_style === "legacy"` means no milestone workspace was ever created.

**How to avoid:** Guard with `if (milestoneWorkspaceExists) { ... }` before any workspace file updates.

**Warning signs:** `complete-milestone` fails on legacy projects after the fix.

### Pitfall 5: Confusing "Update v2.0 Milestone-Scoped ROADMAP" with "Update Root ROADMAP"

**What goes wrong:** The stale ROADMAPs (v1.0, v1.1, v2.0, v3.0 milestone workspace copies) are historical artifacts. Manually updating them is optional polish â€” if done, only update the milestone workspace copies, not the root ROADMAP.md (which is already accurate).

**Why it happens:** Multiple ROADMAP files with the same phase names but different states.

**How to avoid:** Only touch milestone workspace ROADMAPs. Root ROADMAP.md is the source of truth and is already correct for phase/plan-level checkboxes (aside from the plan-level `[ ]` vs `[x]` issue that the code fix addresses going forward).

## Open Questions

1. **Should `milestones/v3.0/` workspace be deleted or corrected?**
   - What we know: Layout style is `legacy` â€” all v3.0 work happens at root `.planning/`. The workspace was scaffolded by `new-milestone` but never used. It contains a stale STATE.md with a contradictory design decision.
   - What's unclear: If a future phase ever switches `layout_style` to `milestone-scoped`, does having this workspace help or confuse? Could be argued either way.
   - Recommendation: Delete the workspace in 16.1 cleanup. If milestone-scoped mode is ever needed, `new-milestone` can re-create it correctly. A stale/contradictory workspace causes more confusion than starting fresh.

2. **How many ROADMAP stale-fix updates are in scope for manual cleanup?**
   - What we know: Multiple milestone workspace ROADMAPs are frozen (v1.0, v1.1, v2.0, v3.0). The code fix for plan-level checkboxes applies to future completions, not retroactively.
   - What's unclear: Is retroactively updating historical milestone ROADMAPs worth the effort? They are archived documents.
   - Recommendation: Only update the v3.0 milestone workspace ROADMAP (if we keep the workspace) and the root ROADMAP's plan-level checkboxes for phases 15 and 16. Historical milestone ROADMAPs (v1.0, v1.1, v2.0) are archived â€” their stale state is acceptable and updating them is cosmetic with no operational impact.

3. **Is FLAW 2 (milestone ROADMAPs not updated during plan execution) in scope for 16.1?**
   - What we know: FLAW 2 requires threading milestone-ROADMAP update calls into `plan-phase`, `execute-phase`, and `phase complete` workflows for milestone-scoped projects. This project runs in `legacy` mode, so the impact is zero for current work.
   - What's unclear: Whether this blocking issue will matter before Phase 17 (migration tool).
   - Recommendation: Defer FLAW 2 to Phase 17. It is architecturally related to migration tooling (which normalizes the file layout) and can be addressed when that plumbing is already being touched.

4. **Should FLAW 6 (`new-milestone` creates workspace that may never be used) be fixed in 16.1?**
   - What we know: The root cause is that `new-milestone` always creates a workspace regardless of `layout_style`. The fix would be to gate workspace creation on `concurrent: true` in config.json, or to set `concurrent: true` automatically.
   - What's unclear: Whether this has caused any actual operational problems beyond the orphaned v3.0 workspace.
   - Recommendation: Defer. FLAW 6 is a design question for the `new-milestone` workflow that should be addressed when the milestone system is revisited. For 16.1, deleting the orphaned v3.0 workspace is sufficient remediation.
