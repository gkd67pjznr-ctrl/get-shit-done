**Frontmatter:**
```yaml
---
name: gsd:fix-debt
description: Route a debt entry through diagnosis and fix execution
argument-hint: "[TD-ID or blank for auto-select]"
```

**Objective section:**
- Describe purpose: orchestrate debt entry from open -> diagnosed -> fixed -> resolved
- State orchestrator role: select entry, spawn debugger, create plan, spawn executor, mark resolved
- Explain why subagent isolation: debugger and executor each burn context independently

**Process section — 8 numbered steps:**

**Step 0: Initialize Context**
- Load state via `node gsd-tools.cjs state load`, extract `commit_docs`
- Resolve models: `resolve-model gsd-debugger --raw` and `resolve-model gsd-executor --raw`

**Step 1: Select Debt Entry**
- If `$ARGUMENTS` matches `TD-\d+` pattern: look up that specific entry via `debt list --raw` + jq select
- If no argument: `debt list --status open --raw` + jq sort by severity (critical=0, high=1, medium=2, low=3) then by date_logged ascending, pick first
- If no open entries exist (`total` is 0): print "No open debt entries found. Nothing to fix." and EXIT
- Display the selected entry to the user: ID, severity, type, component, description, source

**Step 2: Confirm and Mark In-Progress**
- Show the selected entry summary to the user
- If NOT in yolo/autonomous mode: use AskUserQuestion to confirm "Fix this entry? (yes/skip/pick another)"
  - "skip" -> transition to deferred, exit
  - "pick another" -> list all open entries, let user choose, go back to selection
- Transition entry to `in-progress`: `debt resolve --id {ID} --status in-progress`

**Step 3: Assess Diagnosis Need**
- Check entry description richness: if description is substantive (>50 chars, not generic placeholder text)
  - In non-yolo mode: Ask user "Entry has detailed description. Skip diagnosis and go straight to fix plan? (yes/no)"
  - If user says yes or yolo mode with rich description: skip to Step 5 (Plan Fix) using entry description as diagnosis context
- If description is sparse or user says no: proceed to Step 4

**Step 4: Diagnose via gsd-debugger**
- Construct debugger prompt using Pattern 3 from research:
  - Objective: "Investigate debt entry: {entry.id}"
  - Symptoms prefilled from entry fields (description, type, component, source)
  - Mode: `symptoms_prefilled: true`, `goal: find_root_cause_only`
  - Debug file: `.planning/debug/debt-{entry.id}.md`
- Spawn debugger: `Task(prompt=filled_prompt, subagent_type="gsd-debugger", model="{debugger_model}", description="Diagnose {entry.id}: {entry.description}")`
- Handle returns:
  - `## ROOT CAUSE FOUND`: extract root cause, proceed to Step 5
  - `## INVESTIGATION INCONCLUSIVE`: ask user whether to (a) attempt fix anyway, (b) defer entry, (c) add context and retry
    - If defer: `debt resolve --id {ID} --status deferred`, exit
  - `## CHECKPOINT REACHED`: present to user, gather response, spawn continuation

**Step 5: Plan Fix (Inline Mini-Plan)**
- Create directory: `.planning/debug/debt-{entry.id}-fix/`
- Write inline fix plan to `.planning/debug/debt-{entry.id}-fix/01-PLAN.md` with:
  - Frontmatter: `phase: debt-fix-{entry.id}`, `plan: 01`, `type: execute`, `wave: 1`, `depends_on: []`, `files_modified: [entry.component files]`, `autonomous: true`
  - Objective: "Fix debt entry {ID}: {description}. Root cause: {diagnosis or entry description}"
  - Single task with specific fix instructions derived from diagnosis (or entry description if diagnosis skipped)
  - Verification: how to confirm the fix works (test command, grep check, etc.)
  - Success criteria: the debt issue no longer exists
- Scope guard: `files_modified` MUST be limited to the entry's component. If diagnosis reveals broader scope, flag to user before proceeding.

**Step 6: Execute Fix via gsd-executor**
- Spawn executor on the inline plan: `Task(prompt="Execute plan at .planning/debug/debt-{entry.id}-fix/01-PLAN.md. Autonomous. All tasks + SUMMARY + commit.", subagent_type="gsd-executor", model="{executor_model}", description="Fix {entry.id}")`
- Handle executor return:
  - Success: proceed to Step 7
  - Failure/partial: ask user whether to (a) retry with adjustments, (b) defer entry
    - If defer: `debt resolve --id {ID} --status deferred`, exit

**Step 7: Verify Fix**
- Read executor SUMMARY for completion status
- If executor reported all tasks done and tests passing: proceed to Step 8
- If executor reported issues: present to user, offer retry or defer

**Step 8: Mark Resolved**
- Transition entry: `debt resolve --id {ID} --status resolved`
- Print success message: "Debt entry {ID} resolved. {description}"
- Suggest next: "Run `/gsd:fix-debt` again to fix the next entry, or `/gsd:progress` to check milestone status."

**Error handling throughout:**
- If any step fails unexpectedly, transition entry back from `in-progress`:
  - Diagnosis failed -> back to `open` (nothing was attempted)
  - Plan creation failed -> `deferred` with reason
  - Execution failed -> `deferred` (something was attempted)
- NEVER leave an entry stuck in `in-progress` without resolution
