# Phase 3.3: Migration Tool - Research

**Researched:** 2026-02-25
**Domain:** Node.js CLI — filesystem inspection, additive migration, undo manifest
**Confidence:** HIGH

## Summary

Phase 3.3 adds a `migrate.cjs` module and a `migrate` CLI command to `gsd-tools.cjs`. The tool inspects an existing `.planning/` directory against the current GSD spec, reports what would change (`--dry-run`), and applies additive-only structural changes with a recorded undo manifest (`--apply`).

The work is entirely internal to this project's Node.js CLI toolset. No external libraries are needed. All patterns required — `fs`, `path`, JSON output via `output()`, error exit via `error()`, test isolation via `createTempProject()` — are already established in the codebase. The closest analog is `cmdValidateHealth` in `verify.cjs`, which also inspects `.planning/` and produces structured JSON results.

The key design constraint is **additive-only**: `--apply` must never delete or overwrite existing files. It may create missing directories and create missing files (e.g., `config.json` if absent), but it must not modify files that already exist. This makes idempotency trivial: re-running `--apply` finds the same files already present and takes no action.

**Primary recommendation:** Implement `migrate.cjs` as a single self-contained module with two exported functions (`cmdMigrateDryRun`, `cmdMigrateApply`), wire both into the `gsd-tools.cjs` router under `case 'migrate':`, and write tests in a new `tests/migrate.test.cjs` file following the `runGsdTools` + `createTempProject` pattern established in `debt.test.cjs`.

### Pattern 1: Module Structure (mirrors debt.cjs)
**What:** A single `migrate.cjs` with two exported command functions and private helpers.
**When to use:** Every lib module in this project follows this shape.
**Example:**
```javascript
// Source: get-shit-done/bin/lib/debt.cjs (established pattern)
'use strict';

const fs = require('fs');
const path = require('path');
const { output, error, loadConfig, detectLayoutStyle, planningRoot } = require('./core.cjs');

function inspectLayout(cwd) {
  // Returns { layout: 'legacy'|'milestone-scoped'|'uninitialized', changes: [], summary: {} }
}

function cmdMigrateDryRun(cwd, opts, raw) {
  const inspection = inspectLayout(cwd);
  output({ dry_run: true, changes: inspection.changes, summary: inspection.summary }, raw,
    inspection.changes.length === 0 ? 'up-to-date' : 'changes-needed');
}

function cmdMigrateApply(cwd, opts, raw) {
  const inspection = inspectLayout(cwd);
  const actions = [];
  for (const change of inspection.changes) {
    if (change.type === 'create_dir') {
      if (!fs.existsSync(change.path)) {
        fs.mkdirSync(change.path, { recursive: true });
        actions.push({ type: 'created_dir', path: change.path });
      }
    } else if (change.type === 'create_file') {
      if (!fs.existsSync(change.path)) {
        fs.writeFileSync(change.path, change.content, 'utf-8');
        actions.push({ type: 'created_file', path: change.path });
      }
    }
  }
  // Write undo manifest
  const manifestPath = path.join(cwd, '.planning', 'migrate-undo.json');
  const manifest = { applied: new Date().toISOString(), actions };
  fs.writeFileSync(manifestPath, JSON.stringify(manifest, null, 2), 'utf-8');
  output({ applied: true, actions_taken: actions.length, actions, manifest_path: manifestPath }, raw,
    actions.length === 0 ? 'already-up-to-date' : `applied-${actions.length}-changes`);
}

module.exports = { cmdMigrateDryRun, cmdMigrateApply };
```

### Pattern 2: Router Wiring (mirrors debt case block in gsd-tools.cjs)
**What:** A `case 'migrate':` block in the main switch that dispatches `--dry-run` / `--apply`.
**When to use:** Every new command added to gsd-tools.cjs follows this switch-case pattern.
**Example:**
```javascript
// Source: gsd-tools.cjs router pattern (established)
case 'migrate': {
  const subcommand = args[1];
  if (subcommand === '--dry-run' || args.includes('--dry-run')) {
    migrate.cmdMigrateDryRun(cwd, {}, raw);
  } else if (subcommand === '--apply' || args.includes('--apply')) {
    migrate.cmdMigrateApply(cwd, {}, raw);
  } else {
    error('Usage: migrate --dry-run | migrate --apply');
  }
  break;
}
```

Note: The ROADMAP/requirement says `gsd-tools migrate --dry-run` (flag style, not subcommand style). Since flags come after the command name, the router needs to check `args.includes('--dry-run')` not just `args[1]`. The pattern above handles both.

### Pattern 3: Inspection Logic (mirrors cmdValidateHealth in verify.cjs)
**What:** Walk the spec of what `.planning/` should contain, compare to disk, accumulate a `changes` list.
**When to use:** Dry-run inspection phase.
**Example:**
```javascript
// Source: verify.cjs cmdValidateHealth (established pattern for inspection)
function inspectLayout(cwd) {
  const planningDir = path.join(cwd, '.planning');
  const changes = [];

  // Check required dirs
  const requiredDirs = [
    path.join(planningDir, 'phases'),
  ];
  for (const dir of requiredDirs) {
    if (!fs.existsSync(dir)) {
      changes.push({ type: 'create_dir', path: dir, reason: 'Required directory missing' });
    }
  }

  // Check config.json exists
  const configPath = path.join(planningDir, 'config.json');
  if (!fs.existsSync(configPath)) {
    changes.push({
      type: 'create_file',
      path: configPath,
      content: JSON.stringify({ model_profile: 'balanced', commit_docs: true }, null, 2),
      reason: 'config.json missing — required for GSD operations',
    });
  }

  return { layout: detectLayoutStyle(cwd), changes };
}
```

### Pattern 4: Idempotency via Existence Guards
**What:** Every `create_dir` and `create_file` action is gated on `!fs.existsSync(target)`.
**When to use:** Mandatory in `--apply`. This is what makes MIGR-03 trivially true.
**Example:**
```javascript
// Gate every action — this is the idempotency guarantee
if (!fs.existsSync(change.path)) {
  fs.mkdirSync(change.path, { recursive: true });
  actions.push({ type: 'created_dir', path: change.path });
  // else: already exists, skip silently
}
```

### Pattern 5: Undo Manifest
**What:** A JSON file written at `.planning/migrate-undo.json` recording what `--apply` did.
**Format:** `{ applied: ISO8601, actions: [{ type, path }] }`
**Location:** `.planning/migrate-undo.json` (project-level, not milestone-scoped — mirrors DEBT.md placement)
**Note:** The manifest records what was created so it can theoretically be reversed. Since actions are additive-only, "undo" = delete the created files/dirs. The manifest itself is written unconditionally on every `--apply` run (overwritten), so re-runs produce a fresh manifest reflecting the latest execution.

### Anti-Patterns to Avoid
- **Modifying existing files:** MIGR-02 is additive-only. If `config.json` exists but is missing the `quality` section, `--apply` must NOT modify it. That is `config-ensure-section`'s job. Document as "out of scope" in the change report.
- **Destructive actions in undo manifest:** The manifest records what was CREATED. The tool never deletes anything, so the manifest does not need delete actions.
- **Milestone-scoping the migrate command:** The migration operates on project-level `.planning/` structure, not milestone workspace contents. No `--milestone` flag needed.
- **Hardcoding the spec in the router:** The spec (what directories and files should exist) belongs in `migrate.cjs`, not in `gsd-tools.cjs`.

## What the Current Spec Checks (Migration Rules)

Based on `cmdValidateHealth`, `cmdConfigEnsureSection`, and how other modules use `.planning/`, the migration spec should check at minimum:

### Always Required (legacy and milestone-scoped)
| Path | Type | Note |
|------|------|------|
| `.planning/` | dir | Root — if missing, nothing else runs |
| `.planning/phases/` | dir | Phase directory for legacy layout |
| `.planning/config.json` | file | Config — `config-ensure-section` creates it |

### Legacy Layout Checks
When `detectLayoutStyle(cwd) === 'legacy'`:
- `.planning/ROADMAP.md` — warn if missing (cannot create without content)
- `.planning/STATE.md` — warn if missing (cannot create without content)

### Milestone-Scoped Checks
When `detectLayoutStyle(cwd) === 'milestone-scoped'`:
- Milestone workspace dirs (`milestones/vX.Y/phases/`) — can create if missing
- Root `.planning/phases/` still required as fallback

### Cannot Migrate (report as manual action)
Files that require content the tool cannot generate:
- `ROADMAP.md` (requires user-written phase descriptions)
- `STATE.md` (requires current project state)
- `PROJECT.md` (requires user-written description)

The `changes` list should distinguish `create_dir` (automated) from `manual_action` (reported but not applied).

## Common Pitfalls

### Pitfall 1: Checking `args[1] === '--dry-run'` Only
**What goes wrong:** User might write `migrate --dry-run` where `args[1]` is `'--dry-run'`, or they might write `migrate --cwd /some/path --dry-run` where `args[1]` is something else.
**Why it happens:** Router strips `--cwd` before dispatch, but other patterns can still shift arg positions.
**How to avoid:** Use `args.includes('--dry-run')` rather than positional checks. This is consistent with how the `--raw`, `--amend`, and `--archive-phases` flags are handled in gsd-tools.cjs.
**Warning signs:** Tests pass for `migrate --dry-run` but fail for `migrate --cwd /tmp --dry-run`.

### Pitfall 2: Non-idempotent Manifest Writes
**What goes wrong:** If the manifest is appended rather than overwritten, repeated `--apply` runs grow the manifest.
**Why it happens:** Following the debt.cjs `appendFileSync` pattern without realizing it's wrong here.
**How to avoid:** Use `fs.writeFileSync` (not `appendFileSync`) for the manifest. The manifest always reflects the most recent `--apply` run.
**Warning signs:** Manifest has duplicate action entries after second `--apply`.

### Pitfall 3: Confusing Dry-Run Output with Apply Output
**What goes wrong:** Dry-run says "3 changes needed" but apply says "0 changes applied" (or vice versa) because they use different inspection logic.
**Why it happens:** Duplicating the inspection logic instead of sharing it.
**How to avoid:** Extract `inspectLayout(cwd)` as a shared private function called by both `cmdMigrateDryRun` and `cmdMigrateApply`. MIGR-01 and MIGR-02 both need the same change list.
**Warning signs:** Test for dry-run and apply disagree on the same fixture.

### Pitfall 4: Forgetting the `'use strict'` Header
**What goes wrong:** Inconsistency with the rest of the codebase.
**Why it happens:** New file template doesn't include it.
**How to avoid:** Every `lib/*.cjs` file starts with `'use strict';` — copy the debt.cjs header.

### Pitfall 5: Testing Against the Live Project's `.planning/`
**What goes wrong:** Tests pass locally (project is already migrated) but fail in CI or on a fresh checkout.
**Why it happens:** Using `process.cwd()` instead of a `createTempProject()` temp dir.
**How to avoid:** Always use `createTempProject()` or `createConcurrentProject()` from `tests/helpers.cjs` for isolated fixtures.

## Open Questions

1. **What exact checks should define "up to spec"?**
   - What we know: At minimum: `.planning/` exists, `.planning/phases/` exists, `config.json` exists.
   - What's unclear: Should the migration also check for `config.json` having a `quality` section? That overlaps with `config-ensure-section`.
   - Recommendation: Keep migrate's scope narrow — structural directories and the config file's existence only. Do NOT check config section contents; delegate that to `config-ensure-section`.

2. **Should `--apply` create `config.json` with default content or defer to `config-ensure-section`?**
   - What we know: `config-ensure-section` already does this well with user-level defaults.
   - What's unclear: Calling `cmdConfigEnsureSection` internally from `cmdMigrateApply` would be clean but crosses module boundaries.
   - Recommendation: Write minimal `config.json` content (same defaults as `cmdConfigEnsureSection`). OR internally call `config-ensure-section` as a child process. Planner should decide which; both work.

3. **Undo manifest location for milestone-scoped projects?**
   - What we know: DEBT.md is at project-level `.planning/DEBT.md`, not milestone-scoped.
   - Recommendation: Follow the same convention — `migrate-undo.json` at `.planning/migrate-undo.json`.
