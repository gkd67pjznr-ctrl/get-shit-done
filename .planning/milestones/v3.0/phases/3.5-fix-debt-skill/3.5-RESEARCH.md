## Summary

Phase 3.5 is a **Markdown authoring phase**: it delivers one new slash command file (`.claude/commands/gsd/fix-debt.md`) and nothing else — no new `.cjs` modules, no new CLI commands. The skill is an orchestrator that coordinates existing tools: the `debt list` CLI, the `gsd-debugger` agent, the `plan-phase --gaps` workflow, the `execute-phase` workflow, and the `debt resolve` CLI.

The full orchestration sequence is: (1) read DEBT.md via `debt list`, (2) select an entry by ID argument or by priority (critical → high → severity order), (3) transition the entry to `in-progress` via `debt resolve`, (4) spawn `gsd-debugger` with `goal: find_root_cause_only` to get a diagnosis, (5) create a fix plan via `plan-phase --gaps` (or inline mini-plan), (6) execute via `execute-phase`, (7) run verification, and (8) mark the entry `resolved` via `debt resolve`.

All components exist: the debt CLI (Phase 3.2), the debugger agent (pre-existing), the plan-phase/execute-phase workflows (pre-existing). This phase wires them into a single command that a user can invoke as `/gsd:fix-debt [TD-ID]`. The design decision that drives architecture is whether diagnosis is always required or only when there is no description. Given that DEBT.md entries from agent auto-logging (Phase 3.4) already carry rich descriptions, diagnosis depth must be an orchestrator decision, not a rigid gate.

**Primary recommendation:** Author a single slash command file modeled on `gsd:debug` — orchestrator spawns `gsd-debugger` (with `goal: find_root_cause_only`), then hands the diagnosis to `plan-phase --gaps` logic (or creates an inline plan), then runs `execute-phase`, then marks resolved. Keep the orchestrator lean: all intelligence lives in existing subagents.

### Pattern 2: Entry Selection Logic

**What:** Select debt entry by ID argument or by priority fallback.

**When to use:** Whenever FIXD-01 runs without an explicit ID argument.

**Priority order (from DEBT.md schema):**
1. If `$ARGUMENTS` matches `TD-NNN` pattern: use that entry
2. If no argument: `debt list --status open --raw` → sort by severity (critical → high → medium → low) → pick first

**Example selection logic:**
```bash
# Arg-based selection
INIT=$(node /Users/tmac/.claude/get-shit-done/bin/gsd-tools.cjs state load)
COMMIT_DOCS=$(echo "$INIT" | jq -r '.commit_docs')
DEBUGGER_MODEL=$(node /Users/tmac/.claude/get-shit-done/bin/gsd-tools.cjs resolve-model gsd-debugger --raw)

if echo "$ARGUMENTS" | grep -qE '^TD-[0-9]+$'; then
  ENTRY=$(node /Users/tmac/.claude/get-shit-done/bin/gsd-tools.cjs debt list --raw | \
    jq -r --arg id "$ARGUMENTS" '.entries[] | select(.id == $id)')
else
  # Priority-ordered auto-select: critical > high > medium > low
  ENTRY=$(node /Users/tmac/.claude/get-shit-done/bin/gsd-tools.cjs debt list --status open --raw | \
    jq -r '
      .entries |
      sort_by(
        if .severity == "critical" then 0
        elif .severity == "high" then 1
        elif .severity == "medium" then 2
        else 3 end
      ) |
      first
    ')
fi
```

### Pattern 4: Fix Plan Creation

**What:** After diagnosis, create a fix plan. The natural route is via `plan-phase --gaps` but this requires a VERIFICATION.md with gaps — which does not exist for a standalone debt fix. The alternative is an inline plan structure written into a temporary phase directory, then executed.

**Key insight:** Phase 3.5 is NOT creating a new roadmap phase for each debt fix. It creates an **ephemeral repair plan** in the current working context. The plan authoring should be done directly by the orchestrator (or a spawned planner agent) using the diagnosis as input, written to `.planning/debug/debt-{id}-fix/` or a dedicated repair workspace.

**Two viable approaches identified:**

Approach A — Inline mini-plan (simpler):
- Orchestrator writes a minimal PLAN.md to `.planning/debug/debt-{entry.id}-fix/01-PLAN.md`
- Spawns a `gsd-executor` subagent directly on that plan
- No plan-checker loop, no research

Approach B — Route through plan-phase --gaps (full quality):
- Orchestrator creates a temporary VERIFICATION.md-like file with the diagnosis as the gap
- Invokes `plan-phase --gaps` to let the planner create a verified plan
- Then invokes `execute-phase` on the resulting plan

**Recommendation:** Approach A for the MVP (Phase 3.5). A debt fix is one targeted change; plan-phase --gaps overhead (researcher spawn, plan-checker loop) is disproportionate. The planner can always escalate to full plan-phase if the fix is complex.

### Pattern 5: Resolved Status Transition

**What:** After fix is verified, transition debt entry to `resolved`.

**When to use:** After successful execution verification. On user-confirms-fixed checkpoint.

```bash
node /Users/tmac/.claude/get-shit-done/bin/gsd-tools.cjs debt resolve \
  --id "${ENTRY_ID}" \
  --status resolved
```

For deferred (couldn't fix): `--status deferred`

### Anti-Patterns to Avoid

- **Calling `plan-phase --gaps` without a valid VERIFICATION.md:** `plan-phase --gaps` reads VERIFICATION.md for gap context. For debt fixes there is no VERIFICATION.md — the gap context is the debt entry + diagnosis. Don't force the pattern; use inline plan authoring instead.
- **Skipping the in-progress transition:** Without transitioning to `in-progress` before diagnosis, concurrent skill invocations could pick the same entry. Mark it `in-progress` before spawning any subagent.
- **Marking resolved before user verification:** The `gsd:debug` pattern requires human-verify checkpoint. For debt fixes, the executor's built-in quality gates (test_gate, diff_review) substitute for human-verify in yolo mode. In interactive mode, require confirmation.
- **Creating a new roadmap phase:** Debt fixes are not ROADMAP phases. They are ephemeral. Nothing gets added to ROADMAP.md or STATE.md (beyond the debt status update in DEBT.md).
- **Logging a new debt entry if the fix itself encounters issues:** If the fix plan execution defers a sub-issue, that sub-issue should be logged as a new debt entry (WIRE-01 pattern), not as a blocker on resolving the original entry.

## Common Pitfalls

### Pitfall 1: Diagnosis Depth vs Debt Entry Richness

**What goes wrong:** The debt entry already has type, severity, component, description, source_phase, and source_plan. The orchestrator spawns a deep debugger investigation when the fix is obvious from the entry alone.

**Why it happens:** The pattern from `gsd:debug` always does a full investigation. For debt entries created by agent auto-logging (WIRE-01, WIRE-02), the description is already diagnostic (e.g., "Strict mode gate blocked: test_gate failed for auth.cjs — 3 failing assertions").

**How to avoid:** The orchestrator should check entry richness before spawning debugger. If description is substantive (non-empty, non-generic), offer the user the option to skip diagnosis and go straight to plan. If description is sparse, always diagnose.

**Warning signs:** Debugger investigation contradicts the debt entry's own description — the entry already had the answer.

### Pitfall 2: Plan Execution Scope Leak

**What goes wrong:** The inline fix plan modifies files beyond the debt entry's component, creating unexpected changes or regressions.

**Why it happens:** The executor follows deviation rules but the plan author (orchestrator or user) may write a plan too broad for the specific debt issue.

**How to avoid:** The inline plan's `files_modified` list must be scoped to the debt entry's `component` field. If the diagnosis reveals broader scope is needed, the orchestrator should flag this to the user before executing.

**Warning signs:** `files_modified` in the plan includes more than 3-4 files for a single debt entry.

### Pitfall 3: Status Left in in-progress

**What goes wrong:** If execution fails partway through, the debt entry remains `in-progress` with no way to discover it has stalled.

**Why it happens:** The `in-progress` transition happens at step 2 (before diagnosis). If diagnosis, planning, or execution fails, the entry never transitions back to `open` or forward to `resolved`.

**How to avoid:** The orchestrator must handle failures explicitly:
- Diagnosis fails → transition back to `open`, report to user
- Plan creation fails → transition to `deferred`, log reason
- Execution fails → transition to `deferred` (not back to `open` — something was attempted)

**Warning signs:** `debt list --status in-progress` shows entries with old `date_logged` values.

### Pitfall 4: Confusing Debt Fix Phase with Roadmap Phase

**What goes wrong:** The skill creates a new numbered phase in the ROADMAP/STATE planning files, treating a debt fix like a roadmap milestone.

**Why it happens:** The `execute-phase` workflow reads ROADMAP.md for phase context. If the orchestrator passes a fabricated phase number, it will try to find it in ROADMAP.md.

**How to avoid:** The inline fix plan does NOT go through `init execute-phase` with a phase number. The executor is spawned directly on the plan file path, bypassing the ROADMAP lookup. Alternatively, use a reserved namespace like `debt-fix/{entry_id}` that the executor can handle without ROADMAP context.

### Pitfall 5: Multiple Open Entries — Which to Fix

**What goes wrong:** DEBT.md has 10 open entries; the user runs `/gsd:fix-debt` with no argument. The auto-selection picks a medium-severity entry when a critical one exists.

**Why it happens:** Priority ordering must be consistent (critical → high → medium → low). Within the same severity, date_logged order (oldest first) is the natural secondary sort.

**How to avoid:** The selection logic must use a deterministic sort. The research code example above (jq sort_by severity index) is correct. Secondary sort by date_logged ascending ensures oldest debt is addressed first within equal severity.

## Open Questions

1. **Should the orchestrator always spawn the debugger, or offer a "quick fix" path for rich debt entries?**
   - What we know: Entries from agent auto-logging (Phase 3.4) already contain actionable descriptions. Full debugger investigation may be redundant.
   - What's unclear: Whether the orchestrator can reliably assess entry richness programmatically (description length? logged_by field?).
   - Recommendation: Offer two paths in the orchestrator: (A) "I have enough context — write fix plan directly" for entries where description is substantive, (B) "Diagnose first" for entries with sparse descriptions. Let the user confirm which path when not in yolo mode.

2. **Where should the inline fix plan live?**
   - What we know: `.planning/debug/` exists for debug session files. Using a `debt-{id}-fix/` subdirectory there keeps debt-related ephemera in one place.
   - What's unclear: Whether `execute-plan` workflow's `init execute-phase` call will gracefully handle a non-ROADMAP phase path.
   - Recommendation: Write the plan to `.planning/debug/debt-{id}-fix/01-PLAN.md`. Spawn `gsd-executor` directly on the file path rather than routing through `init execute-phase`. This avoids the ROADMAP lookup entirely.

3. **What happens when no open debt entries exist?**
   - What we know: `debt list --status open` returns `{ entries: [], total: 0 }` gracefully.
   - What's unclear: Nothing — this is trivially handled: print "No open debt entries" and exit.
   - Recommendation: Check `total` from debt list output first; exit cleanly if 0.
