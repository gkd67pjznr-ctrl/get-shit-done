## Summary

Phase 3.4 is entirely a **Markdown authoring phase**: no new `.cjs` modules, no new CLI commands. The deliverables are edits to two agent instruction files (`gsd-executor.md` and `gsd-verifier.md`) that add debt-logging behavior at existing, well-defined hook points.

The executor has two places where issues become "unfixable in the current plan": (1) the FIX ATTEMPT LIMIT (3 failed auto-fix attempts on a task, issues moved to SUMMARY "Deferred Issues"), and (2) strict-mode `blocked` gate outcomes from the quality sentinel. The verifier has one place: Step 7b quality findings that are WARN (standard) or FAIL (strict) after all sub-checks complete.

Quality-level gating already exists in both agents (they both read `quality.level` at sentinel entry). Adding debt logging follows the exact same conditional pattern — `fast` means no logging, `standard` means log critical/high severity only, `strict` means log all severities. This is consistent with the existing gate behavior matrix.

**Primary recommendation:** Add a `<debt_logging>` subsection inside `<quality_sentinel>` in the executor, and add a `<debt_logging>` step after Step 7b in the verifier. Both sections read quality level, check severity gating, and call `gsd-tools debt log` with provenance fields populated from already-in-scope variables.

## Architecture Patterns

### Where to Hook in gsd-executor.md

The executor's `<quality_sentinel>` section already contains:

1. **FIX ATTEMPT LIMIT** (in `<deviation_rules>`, not technically quality_sentinel): After 3 auto-fix attempts fail, issues go to SUMMARY "Deferred Issues" and execution continues. This is the "unfixable in current plan" event. The debt log call belongs here, gated on quality level and severity.

2. **`blocked` gate outcomes** (in quality sentinel post-task): In strict mode, a `test_gate|blocked` or `diff_review|blocked` outcome prevents the commit. This is a second "unfixable" signal. The debt log call belongs at the point where a `blocked` outcome is recorded.

The natural insertion point for both is a new `## Debt Auto-Log Protocol` subsection inside `<quality_sentinel>`, with two triggers:

**Trigger A — FIX ATTEMPT LIMIT reached:**
```bash
# In <deviation_rules>, after "STOP fixing — document remaining issues in SUMMARY.md"
QUALITY_LEVEL=$(node /Users/tmac/.claude/get-shit-done/bin/gsd-tools.cjs config-get quality.level 2>/dev/null || echo "fast")
if [ "$QUALITY_LEVEL" != "fast" ]; then
  # Determine severity of deferred issue (critical/high/medium/low)
  # standard: log critical and high only; strict: log all
  ISSUE_SEVERITY="medium"  # or derive from context
  SHOULD_LOG=false
  if [ "$QUALITY_LEVEL" = "strict" ]; then SHOULD_LOG=true; fi
  if [ "$QUALITY_LEVEL" = "standard" ] && ([ "$ISSUE_SEVERITY" = "critical" ] || [ "$ISSUE_SEVERITY" = "high" ]); then SHOULD_LOG=true; fi
  if [ "$SHOULD_LOG" = "true" ]; then
    node /Users/tmac/.claude/get-shit-done/bin/gsd-tools.cjs debt log \
      --type code \
      --severity "$ISSUE_SEVERITY" \
      --component "${TASK_COMPONENT}" \
      --description "${DEFERRED_ISSUE_DESCRIPTION}" \
      --logged-by executor \
      --source-phase "${PHASE}" \
      --source-plan "${PLAN}"
  fi
fi
```

**Trigger B — blocked gate outcome (strict mode only):**
```bash
# After recording GATE_OUTCOMES+=("|test_gate|blocked|...")
if [ "$QUALITY_LEVEL" = "strict" ]; then
  node /Users/tmac/.claude/get-shit-done/bin/gsd-tools.cjs debt log \
    --type test \
    --severity high \
    --component "${TASK_FILES}" \
    --description "Strict mode gate blocked: ${GATE_DETAIL}" \
    --logged-by executor \
    --source-phase "${PHASE}" \
    --source-plan "${PLAN}"
fi
```

### Where to Hook in gsd-verifier.md

The verifier's Step 7b already collects all WARN/FAIL findings into `FINDINGS_DUPLICATION`, `FINDINGS_ORPHANED`, `FINDINGS_MISSING_TESTS` arrays. After writing the "Step 7b: Quality Findings" section to VERIFICATION.md, add a debt auto-logging step:

```bash
# After Step 7b section written to VERIFICATION.md
QUALITY_LEVEL=$(node /Users/tmac/.claude/get-shit-done/bin/gsd-tools.cjs config-get quality.level 2>/dev/null || echo "fast")
if [ "$QUALITY_LEVEL" != "fast" ]; then
  for finding in "${ALL_FINDINGS[@]}"; do
    # Parse severity from finding string (WARN = high, FAIL = high in standard; all in strict)
    FINDING_SEVERITY=$(echo "$finding" | grep -o "WARN\|FAIL")
    SHOULD_LOG=false
    if [ "$QUALITY_LEVEL" = "strict" ]; then SHOULD_LOG=true; fi
    if [ "$QUALITY_LEVEL" = "standard" ] && [ "$FINDING_SEVERITY" = "FAIL" ]; then SHOULD_LOG=true; fi
    # Note: standard mode WARNs from verifier → severity "high" (structural quality debt)
    if [ "$QUALITY_LEVEL" = "standard" ] && [ "$FINDING_SEVERITY" = "WARN" ]; then SHOULD_LOG=true; fi
    if [ "$SHOULD_LOG" = "true" ]; then
      FINDING_TYPE=$(echo "$finding" | grep -oE "duplication|orphaned|missing.test" | head -1 || echo "quality")
      node /Users/tmac/.claude/get-shit-done/bin/gsd-tools.cjs debt log \
        --type "$FINDING_TYPE" \
        --severity high \
        --component "${FINDING_FILE}" \
        --description "${FINDING_TEXT}" \
        --logged-by verifier \
        --source-phase "${PHASE_NUM}" \
        --source-plan "${PLAN_REFERENCE}"
    fi
  done
fi
```

### Anti-Patterns to Avoid

- **Logging in fast mode:** The quality sentinel already gates all steps on `fast`. Debt logging must follow the same pattern — check level first, bail on `fast`.
- **Double-logging:** The executor logs issues it can't fix during execution. The verifier logs issues it finds during phase review. These are different events and should not deduplicate — DEBT.md IDs prevent confusion.
- **Hardcoding severity:** The severity of executor-deferred issues is context-dependent (the agent must assess severity of the deferred issue, not default to "low"). The verifier findings follow a fixed mapping: verifier WARN → high, verifier FAIL → high (strict) or critical (if blocking).
- **Missing provenance:** WIRE-04 requires source_phase and source_plan. Both agents already have PHASE/PLAN variables in scope — these must always be passed to `debt log`.
- **Logging gaps_found truths:** WIRE-02 targets Step 7b **quality dimension** findings (duplication, orphaned exports, missing tests), NOT the goal-achievement gaps from Steps 3-5 (VERIFIED/FAILED truths). The planner uses gaps for re-planning; debt is for structural quality issues that survived the phase.

## Common Pitfalls

### Pitfall 1: Adding Debt Logging to the Wrong Verifier Section

**What goes wrong:** Adding debt logging to Steps 3-5 (truth verification) instead of Step 7b (quality dimensions), causing goal gaps to be logged as debt entries.

**Why it happens:** "unresolved quality dimension findings" (WIRE-02) is easy to conflate with "gaps_found" truths from goal verification. They are different: goal gaps go through the planner for re-execution; quality dimension findings (Step 7b) are structural issues that persist after the phase completes.

**How to avoid:** The debt hook in the verifier belongs AFTER Step 7b writes its section to VERIFICATION.md, and only operates on `FINDINGS_DUPLICATION`, `FINDINGS_ORPHANED`, `FINDINGS_MISSING_TESTS` — not on the `gaps:` YAML frontmatter items.

**Warning signs:** If the instruction says "log every gaps_found truth as debt", that's the wrong hook.

### Pitfall 2: Severity Mapping Ambiguity

**What goes wrong:** Unclear mapping between quality sentinel `outcome` values and DEBT.md `severity` values (critical/high/medium/low). The sentinel uses `blocked`, `warned`, `passed`, `skipped` — the debt schema uses different vocabulary.

**Why it happens:** The two schemas were designed independently. The planner must define a mapping explicitly.

**How to avoid:** Define the severity mapping in the instruction text:
- Executor FIX ATTEMPT LIMIT deferred issues: severity is assessed contextually by the agent (is this a security issue? → critical; broken test coverage? → high; style debt? → medium)
- Executor `blocked` gate (strict): severity = high (gates block for important reasons)
- Verifier WARN findings: severity = high (structural quality issues)
- Verifier FAIL findings: severity = high (strict-mode enforcement failures)

### Pitfall 3: Quality Level Checked Once vs Per-Finding

**What goes wrong:** Quality level is read once per sentinel entry (correct for gates) but debt logging needs per-finding severity gating (should I log this specific finding?).

**Why it happens:** The "standard=critical/high only" requirement means the quality level check alone isn't sufficient — you also need to check the finding's severity against the quality level threshold.

**How to avoid:** The gating logic is two-part: (1) if quality_level is fast, skip entirely; (2) if quality_level is standard, only log findings of severity critical or high; (3) if quality_level is strict, log all findings.

### Pitfall 4: Plan Reference for Verifier

**What goes wrong:** The verifier operates at phase level, not plan level, so `PLAN` may not be defined when the verifier runs. `--source-plan` could be empty or wrong.

**Why it happens:** The verifier agent does not execute individual plans — it verifies the entire phase outcome. It reads plan files but doesn't have a single "current plan" the way the executor does.

**How to avoid:** The verifier should derive `--source-plan` from the list of plans it verified (e.g., `3.4-01,3.4-02` for multi-plan phases, or just `3.4-01` for single-plan). Alternatively, use `"phase-verification"` as the source_plan value for verifier-originated entries — this is unambiguous and consistent.

### Pitfall 5: Breaking Existing Tests

**What goes wrong:** If the plan accidentally touches `.cjs` files when only `.md` files needed to change, tests may fail.

**Why it happens:** If a planner adds "test the agent wiring" tasks, there's no `.cjs` to test — the behavior is in Markdown instructions. Tests for Markdown instruction changes would need to run an actual agent, which is out of scope.

**How to avoid:** This phase has zero `.cjs` changes. The plan should contain only `type="auto"` tasks editing `.md` files. The test_exemptions config already exempts `.md` files from test gate requirements.

## Open Questions

1. **Severity assessment for executor-deferred issues**
   - What we know: The executor reaches FIX ATTEMPT LIMIT when 3 auto-fix attempts fail. The nature of the issue varies (broken tests, unresolved imports, style violations).
   - What's unclear: Should the agent assess severity contextually (e.g., "broken tests = high, style = low"), or use a fixed severity for all executor-deferred items?
   - Recommendation: Contextual assessment by the executor agent is more accurate. The instruction should say "assess severity based on issue type: security/correctness → critical, broken tests → high, code quality → medium, style → low." This keeps the debt signal meaningful.

2. **Verifier source_plan value**
   - What we know: The verifier runs at phase level, not plan level. The `--source-plan` field exists in the debt schema.
   - What's unclear: Should verifier entries use `"phase-verification"` as source_plan, or derive plan names from the plans it reviewed?
   - Recommendation: Use `"phase-verification"` as a fixed string for verifier-originated entries. It's unambiguous, clearly distinguishes verifier debt from executor debt, and requires no derivation logic.

3. **Deduplication across executor and verifier**
   - What we know: The executor could log "missing tests for foo.cjs" during execution, and the verifier could log the same finding during verification.
   - What's unclear: Should there be deduplication logic?
   - Recommendation: No deduplication needed. DEBT.md's TD-NNN IDs distinguish entries. The provenance fields (`logged_by: executor` vs `logged_by: verifier`) and `source_plan` values make origin clear. Phase 3.5 (/gsd:fix-debt) will handle deduplication when selecting entries to fix.
