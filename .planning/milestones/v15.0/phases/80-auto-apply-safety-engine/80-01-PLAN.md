# Plan 80-01: Auto-Apply Engine — Core Module with Five Safety Gates

## Goal

Create `.claude/hooks/lib/auto-apply.cjs` — the auto-apply engine that evaluates pending skill refinement suggestions through five ordered safety gates and calls `acceptSuggestion()` when all gates pass. Gate failures leave the suggestion as pending and write a "skipped" entry to the audit log. The module exports `runAutoApply({ cwd })` for use by the SessionEnd hook.

## Context

- The SessionEnd hook at `.claude/hooks/gsd-analyze-patterns.cjs` calls `analyzePatterns()` then will call `runAutoApply()` (wired in Plan 80-02).
- `acceptSuggestion({ suggestionId, cwd })` in `.claude/hooks/lib/refine-skill.cjs` handles the full apply flow — read suggestion, modify SKILL.md, git commit, append SKILL-HISTORY.md, retire corrections.
- `computeAttributionConfidence()` in `get-shit-done/bin/lib/skill-metrics.cjs` returns `'high'` / `'medium'` / `'low'` per skill. Quality gate uses this.
- Config lives at `.planning/config.json`. The `auto_apply` key is at `adaptive_learning.auto_apply` (boolean, default false when absent).
- Audit log is `.planning/patterns/auto-applied.jsonl` (append-only, no rotation needed).
- Silent failure convention: all hook utilities catch errors and never throw — the engine must exit cleanly even on unexpected errors.

## Controversial Categories

Auto-apply is skipped for these correction categories (controversial flag check, built into confidence gate step):
- `code.over_engineering`
- `code.scope_drift`
- `process.planning_error`

## Gate Execution Order (fail-fast, cheapest first)

1. **CONFIG GATE** — `adaptive_learning.auto_apply === true` in config.json
2. **RATE GATE** — No entry in `auto-applied.jsonl` with matching skill name and `action: "applied"` within the last 7 days
3. **QUALITY GATE** — Skill attribution confidence is NOT `'high'` (read via `computeAttributionConfidence()` from skill-metrics.cjs)
4. **CONFIDENCE GATE** — `suggestion.confidence > 0.95` AND category is not controversial
5. **SIZE GATE** — Content change between current SKILL.md and what would be written is < 20% (character-count based)

<action>
Inside `auto-apply.cjs`, implement both gate helpers and wire them into the per-suggestion pipeline.

**RATE GATE — `checkRateGate(skillName, cwd)`:**

1. Reads `.planning/patterns/auto-applied.jsonl` line by line. If the file does not exist, return `{ pass: true }`.
2. Parses each JSONL line. Skip malformed lines silently.
3. For each entry where `entry.skill === skillName` and `entry.action === 'applied'`:
   - Parse `entry.timestamp` as a Date.
   - If `(now - entryTimestamp) < 7 * 24 * 60 * 60 * 1000`, return `{ pass: false, reason: 'rate_limit', last_applied: entry.timestamp }`.
4. If no qualifying entry found, return `{ pass: true }`.

**QUALITY GATE — `checkQualityGate(skillName, cwd)`:**

1. Reads `.planning/patterns/skill-metrics.json` from `cwd`. If missing or malformed, return `{ pass: true }` (fail-open: if metrics unavailable, do not block).
2. Looks up the entry for `skillName` in the metrics object/array. Check the actual file format at `.planning/patterns/skill-metrics.json` to determine the correct access pattern.
3. Calls `computeAttributionConfidence(correctionCount, sessionCount)` from `get-shit-done/bin/lib/skill-metrics.cjs` with the skill's correction and session counts from the metrics file, OR reads the pre-computed quality label directly if available.
4. If the resolved quality label is `'high'`, return `{ pass: false, reason: 'quality_gate', quality: 'high' }`.
5. Otherwise return `{ pass: true }`.

**Wiring:** In the gate pipeline within `runAutoApply`, for each pending suggestion call `checkRateGate` first (using `suggestion.target_skill`), then `checkQualityGate`. On any gate failure, call `appendAuditEntry` with a "skipped" entry (see T4 for `appendAuditEntry` definition — stub it as a no-op in this task if T4 is not yet complete) and continue to the next suggestion.

<action>
Inside `auto-apply.cjs`, implement both gate helpers and wire them into the per-suggestion pipeline after the RATE and QUALITY gates.

**CONFIDENCE GATE — `checkConfidenceGate(suggestion)`:**

1. Checks `suggestion.confidence`. If `suggestion.confidence` is not a number or is `<= 0.95`, return `{ pass: false, reason: 'confidence_too_low', confidence: suggestion.confidence }`.
2. Checks `suggestion.category` against the controversial list: `['code.over_engineering', 'code.scope_drift', 'process.planning_error']`. If the category matches, return `{ pass: false, reason: 'controversial_category', category: suggestion.category }`.
3. Otherwise return `{ pass: true }`.

**SIZE GATE — `checkSizeGate(suggestion, cwd)`:**

1. Read the current content of the skill file at `.claude/skills/{suggestion.target_skill}/SKILL.md` from `cwd`. If the file does not exist, return `{ pass: false, reason: 'skill_missing' }`.
2. Simulate the content that `acceptSuggestion` would write by constructing the new bullets from `suggestion.sample_corrections` filtered and formatted exactly as done in `refine-skill.cjs` lines 38-55:
   - Filter out empty strings.
   - Format as `- [${suggestion.category}] ${correction}`.
   - If the skill already has a `## Learned Patterns` section, calculate added length as the sum of bullet character lengths (plus newlines). If not, add the section header plus bullets.
3. New content length = original content length + added content length.
4. Change percent = `added_length / original_length` (expressed as a fraction 0..1).
5. If change percent >= 0.20, return `{ pass: false, reason: 'size_too_large', change_percent: changePercent }`.
6. Otherwise return `{ pass: true, change_percent: changePercent }`.

**Wiring:** Call `checkConfidenceGate` after QUALITY GATE, then `checkSizeGate` after CONFIDENCE GATE. On any gate failure, call `appendAuditEntry` with a "skipped" entry and continue.

```json
{
  "action": "skipped",
  "suggestion_id": "<string>",

  "gate": "<gate name: config|rate|quality|confidence|size>",
  "reason": "<gate failure reason string>",
  "timestamp": "<ISO string>"
}
```

After `acceptSuggestion()` returns `{ ok: true }`:
1. Capture the commit SHA: `const commitSha = execSync('git rev-parse HEAD', { cwd }).toString().trim()`. Note: this assumes `acceptSuggestion` completes its commit synchronously before returning. 2. Capture the before/after diff: `const beforeDiff = execSync('git diff HEAD~1 -- ' + skillPath, { cwd }).toString()`. If the execSync call fails (e.g., shallow clone with no parent), catch the error and set `beforeDiff` to `''`.
3. Set `reversal_instructions` to the string `"git revert " + commitSha`.

Require `execSync` from `child_process` at the top of the file.

Complete the gate pipeline in `runAutoApply`:
1. For each pending suggestion, run gates in order: CONFIG (already checked globally) → RATE → QUALITY → CONFIDENCE → SIZE.
2. If any gate fails: call `appendAuditEntry` with a "skipped" entry, increment skipped count, continue to next suggestion.
3. If all gates pass: call `acceptSuggestion({ suggestionId: suggestion.id, cwd })`. If result is `{ ok: true }`: capture commit SHA and diff, call `appendAuditEntry` with "applied" entry, increment applied count. If result is not ok: write a "skipped" entry with `gate: 'apply'` and `reason: result.reason`.
