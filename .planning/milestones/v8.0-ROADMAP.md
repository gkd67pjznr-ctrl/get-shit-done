# Roadmap — Milestone v8.0: Close the Loop

*Created: 2026-04-02*

## Overview

Two systems are built but not working: the skill observation feedback loop (captures corrections but never refines skills) and the quality sentinel gates (documented in the executor but never fire during execution). This milestone closes the gap — wiring analysis triggers and suggestion surfacing for the skill loop, and moving gates from agent instructions to deterministic hooks so they actually run. Both systems finish with end-to-end verification that real data flows.

## Phases

- [x] **Phase 33: Skill Loop Wiring** - Trigger pattern analysis automatically and surface pending suggestions at session start
- [x] **Phase 34: Skill Refinement** - Implement the suggestion-to-SKILL.md modification flow and verify the full feedback loop
- [x] **Phase 35: Gate Enforcement** - Move quality gates from agent instructions to deterministic hooks/wrappers that fire and persist real data
- [x] **Phase 36: Gate Observability** - Confirm real gate data flows to the dashboard and the full gate cycle is verifiable

## Phase Details

### Phase 33: Skill Loop Wiring
**Goal**: Pattern analysis runs automatically and pending suggestions are surfaced to the user at session start
**Depends on**: Nothing (first phase)
**Requirements**: SKILL-01, SKILL-02, SKILL-03
**Success Criteria** (what must be TRUE):
  1. After a session ends, `analyze-patterns.cjs` runs without any manual invocation
  2. `scan-state.json` contains a watermark timestamp after analysis completes
  3. When `suggest_on_session_start: true`, pending entries from `suggestions.json` are presented to the user at the start of a session
  4. Surfaced suggestions show the target skill name and the proposed change so the user can act on them

### Phase 34: Skill Refinement
**Goal**: Accepted suggestions modify the target SKILL.md and are committed; the full correction-to-skill loop is verified end-to-end
**Depends on**: Phase 33
**Requirements**: SKILL-04, SKILL-05, SKILL-06
**Success Criteria** (what must be TRUE):
  1. Accepting a suggestion causes the content of the target `.claude/skills/*/SKILL.md` file to be updated with the suggested change
  2. The accepted suggestion is removed from `suggestions.json` after the skill file is updated and committed
  3. Dismissing a suggestion removes it from `suggestions.json` without modifying any skill file
  4. The next session after a skill refinement loads the updated SKILL.md content (refined skill is active)
  5. A full loop can be demonstrated: an existing correction entry leads to a detectable suggestion, which is accepted, which modifies the skill, which is loaded in the following session

- [x] 34-02: End-to-end verification — trace existing correction data through analysis, surfacing, refinement, and confirm skill loads in next session

### Phase 35: Gate Enforcement
**Goal**: Quality gates execute deterministically via hooks or wrappers, respect quality level config, and persist real entries to gate-executions.jsonl
**Depends on**: Nothing (independent of skill loop)
**Requirements**: GATE-01, GATE-02, GATE-03, GATE-04
**Success Criteria** (what must be TRUE):
  1. Quality gates run during task execution without depending on the executor agent choosing to follow gate instructions
  2. The gate execution approach (hooks, orchestrator wrapper, or hybrid) is decided, documented, and implemented
  3. At least one gate execution produces a real entry in `gate-executions.jsonl` with actual timestamps and outcomes (not synthetic data)
  4. Gates are skipped entirely when `quality.level` is `fast`, warn-only when `standard`, and blocking when `strict`
**Plans**: TBD

- [x] 35-02: Wire write-gate-execution.cjs into the enforcement path so real gate outcomes are persisted with quality-level filtering

### Phase 36: Gate Observability
**Goal**: Real gate execution data flows to the dashboard Gate Health page and running a quick task produces verifiable gate entries
**Depends on**: Phase 35
**Requirements**: GATE-05, GATE-06
**Success Criteria** (what must be TRUE):
  1. The dashboard Gate Health page displays data sourced from real `gate-executions.jsonl` entries (not synthetic attribution data)
  2. Running `/gsd:quick` produces one or more entries in `gate-executions.jsonl` that can be verified immediately after the task completes
  3. Gate Health page metrics (firing rates, outcome distribution) reflect the newly created entries within one dashboard refresh
