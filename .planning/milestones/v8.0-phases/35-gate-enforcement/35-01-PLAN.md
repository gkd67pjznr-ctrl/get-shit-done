# Plan 35-01: Decide Gate Enforcement Approach and Implement the Mechanism

## Goal

Quality gates must execute deterministically during task execution without depending on the executor agent choosing to follow gate instructions. This plan decides the enforcement approach, documents it, and implements the mechanism.

## Context

### Current State

- Five quality sentinel gates exist: `codebase_scan`, `context7_lookup`, `test_baseline`, `test_gate`, `diff_review`
- Gates are currently described in the executor agent instructions as steps the agent should follow — they are advisory, not enforced
- `write-gate-execution.cjs` exists at `.claude/hooks/lib/write-gate-execution.cjs` and is fully functional (validates entries, writes to `gate-executions.jsonl`) but nothing calls it during real execution
- `gate-attribution.jsonl` shows all gates have `passed:0, warned:0, blocked:0, skipped:0` — confirming gates have never fired
- Config at `.planning/config.json` has `quality.level: "standard"` — this is the active quality level
- The hook infrastructure in `.claude/settings.json` uses: `PreToolUse[Bash]`, `PostToolUse[Write|Edit|Bash]`, `SessionStart`, `SessionEnd`

### Decision: What Approach to Use

**Three options were considered (documented in STATE.md as deferred decision):**

1. **Hooks-only**: Intercept tool calls via Claude's hook system (PreToolUse/PostToolUse). Gates fire on specific tool patterns (e.g., `test_gate` on PostToolUse Bash when the command contains `npm test` or `vitest`). Deterministic, cannot be skipped by the agent.

2. **Orchestrator wrapper**: A script called before each task block that runs all applicable gates and blocks progression if `strict` mode fails. Requires the executor to call the wrapper — still depends on executor cooperation.

3. **Hybrid**: Gate-runner library (`gate-runner.cjs`) invoked from hooks for tool-level gates (test detection via Bash PostToolUse, write detection via Write PostToolUse), plus a SessionEnd gate summary.

**Decision: Hooks approach (Option 1), implemented via a shared `gate-runner.cjs` library called from a new `gsd-run-gates.cjs` hook.**

Rationale: Hooks are the only mechanism that fires without executor cooperation. PostToolUse on Bash catches test runs; PostToolUse on Write catches file-write completion (suitable for triggering `diff_review`). The gate runner can synthesize `codebase_scan` and `context7_lookup` from session context embedded in the hook input.

**Gate-to-hook mapping:**

| Gate | Hook Type | Trigger Pattern | Quality Behavior |
|------|-----------|-----------------|-----------------|
| `test_gate` | PostToolUse[Bash] | command contains `npm test`, `vitest`, `node --test`, or `npx vitest` | fast=skip, standard=warn-only on non-zero exit, strict=block on non-zero exit |
| `test_baseline` | PostToolUse[Bash] | same as test_gate; fires when no prior test run in session | fast=skip, standard=warn, strict=block |
| `diff_review` | PostToolUse[Write] | any Write tool call that creates/modifies `.ts`, `.js`, `.cjs`, `.mjs` files | fast=skip, standard=warn on file count >10, strict=block |
| `codebase_scan` | PostToolUse[Bash] | Glob/Grep command (heuristic: command contains `grep` or `find`) | fast=skip, standard=warn if no search detected in session, strict=block |
| `context7_lookup` | PostToolUse[Bash] | Bash command indicates library usage (heuristic: package names in command) | fast=skip, standard=warn, strict=block |

**Simplification for this phase:** Implement `test_gate`, `test_baseline`, and `diff_review` gates (the three most valuable). The `codebase_scan` and `context7_lookup` gates require session-level state tracking that adds complexity. Plan 35-02 will add write-gate-execution.cjs wiring. Phase 36 validates real data flows to the dashboard.

### Files to Create

- `.claude/hooks/lib/gate-runner.cjs` — shared library with gate detection logic and quality-level filtering
- `.claude/hooks/gsd-run-gates.cjs` — hook script called by PostToolUse Bash and PostToolUse Write; reads stdin, delegates to gate-runner, exits 0 always

Create `/Users/tmac/Projects/gsdup/.claude/hooks/lib/gate-runner.cjs` as a CJS module. File header comment: `// Gate runner library: detects gate triggers from hook input and returns gate evaluation results.`

Zero external dependencies — built-in Node.js only plus `require('./write-gate-execution.cjs')`.

**Exports:**

**`getQualityLevel(cwd)`** — reads `.planning/config.json`, returns `'fast'`, `'standard'`, or `'strict'`. Returns `'fast'` on any error.

**`getPhaseAndPlan(cwd)`** — reads the active `STATE.md` (milestone-scoped if available, same logic as in `gsd-correction-capture.js`). Returns `{ phase: number, plan: string }`. Phase from `Phase:` line, plan from `Plan:` line (parse `N of M in current phase` to extract N). Returns `{ phase: 0, plan: '0' }` on error.

**`detectGate(toolName, toolInput)`** — pure function, no I/O. Returns `null` if no gate triggered, or `{ gate: string, outcome: 'passed'|'warned'|'blocked'|'skipped', detail?: string }` if a gate fires.

Gate detection rules (applied in order, return first match):

1. If `toolName === 'Bash'`:
   - Read `toolInput.command` (string).
   - If command matches `/npm\s+test|vitest|node\s+--test|npx\s+vitest/`, this is a **test run**. The gate is `test_gate`. `outcome` is determined by `toolInput` — note: hook input for PostToolUse includes `tool_response`. If `tool_response` is available and `tool_response.exit_code` is non-zero (or `tool_response.content` contains `failing` or `Error` or `FAIL`), outcome is `'warned'` (standard) or `'blocked'` (strict). If no failure signals, outcome is `'passed'`. Return `{ gate: 'test_gate', rawOutcome: 'check_response', detail: 'test run detected' }`.
   - Otherwise return `null`.

2. If `toolName === 'Write'`:
   - Read `toolInput.file_path` or `toolInput.path` (string).
   - If the file path ends in `.ts`, `.tsx`, `.js`, `.cjs`, `.mjs`, this is a **code write**. Gate is `diff_review`. Return `{ gate: 'diff_review', rawOutcome: 'passed', detail: 'code file written' }`.
   - Otherwise return `null`.

3. Otherwise return `null`.

**`applyQualityLevel(rawOutcome, qualityLevel)`** — maps raw gate results to final outcomes:
- If `qualityLevel === 'fast'`, always return `'skipped'`
- If `qualityLevel === 'standard'`, map: `'passed'` → `'passed'`, `'blocked'` → `'warned'`, `'warned'` → `'warned'`, `'skipped'` → `'skipped'`
- If `qualityLevel === 'strict'`, map: `'passed'` → `'passed'`, `'blocked'` → `'blocked'`, `'warned'` → `'warned'`, `'skipped'` → `'skipped'`

**`evaluateGate(toolName, toolInput, toolResponse, options)`** — main function called by the hook:
- `options`: `{ cwd: string, sessionId: string, taskIndex?: number }`
- Calls `detectGate(toolName, toolInput)` — if null, return `{ evaluated: false, reason: 'no_gate_triggered' }`
- Reads quality level via `getQualityLevel(cwd)`. If `'fast'`, return `{ evaluated: false, reason: 'fast_mode_skip' }`
- Gets phase/plan via `getPhaseAndPlan(cwd)`
- For `test_gate`: parse `toolResponse` (string or object) to determine if tests passed or failed:
  - Check if `toolResponse` contains the strings `failing`, ` fail`, `FAIL`, `Error:` or exit code patterns like `exit code 1`; if any match → raw outcome = `'blocked'`; else → raw outcome = `'passed'`
- For `diff_review`: raw outcome is always `'passed'` (the gate fires and records that a code file was written)
- Apply `applyQualityLevel(rawOutcome, qualityLevel)` to get final outcome
- Build entry object: `{ gate, task: options.taskIndex || 0, outcome, quality_level: qualityLevel, phase: phaseAndPlan.phase, plan: phaseAndPlan.plan, timestamp: new Date().toISOString(), detail: detection.detail || '' }`
- Return `{ evaluated: true, gate, outcome, entry }`

Wrap everything in try/catch — return `{ evaluated: false, reason: 'error', error: e.message }` on any unhandled error.

Do not call `writeGateExecution` from this library — that is done in the hook script. This library only evaluates.

Create `/Users/tmac/Projects/gsdup/.claude/hooks/gsd-run-gates.cjs` as a CJS script. File header comment: `// PostToolUse hook: evaluates quality gates on Bash test runs and Write code file operations.`

This script reads stdin JSON with the PostToolUse hook payload:
```
{ session_id, cwd, tool_name, tool_input, tool_response }
```

Logic:

1. Read stdin fully; parse JSON. Silent failure on parse error — exit 0.
2. Extract `session_id`, `cwd`, `tool_name`, `tool_input`, `tool_response` from parsed input. Default `cwd` to `process.cwd()`.
3. Require `gate-runner.cjs` from the same `lib/` directory: `require(path.join(cwd, '.claude/hooks/lib/gate-runner.cjs'))`. If this fails (e.g., file not found), exit 0 silently.
4. Call `evaluateGate(tool_name, tool_input, tool_response, { cwd, sessionId: session_id, taskIndex: 0 })`.
5. If `result.evaluated === false`, exit 0. No output.
6. If `result.evaluated === true`:
   - Require `write-gate-execution.cjs` from `lib/`: `require(path.join(cwd, '.claude/hooks/lib/write-gate-execution.cjs'))`.
   - Call `writeGateExecution(result.entry, { cwd })`.
   - If `process.env.DEBUG_GATE_RUNNER` is set, write JSON summary to stdout: `JSON.stringify({ gate: result.gate, outcome: result.outcome, written: writeResult.written })`.
7. Always exit 0 — hook must never block execution.

The entire main body must be wrapped in a top-level try/catch that exits 0 on any error.
</action>
