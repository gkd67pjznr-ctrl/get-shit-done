## The Problem

The planner starts from scratch every time. After 15 milestones and 80+ plans, the system has executed hundreds of task structures â€” CLI subcommand wiring, test-first development, hook integration, JSONL persistence, dashboard page creation. Each time a new phase needs one of these patterns, the planner agent reinvents the task breakdown from zero.

This is wasteful and inconsistent. Plan 42-01 (skill-scorer) had nearly identical structure to 41-01 (skill-metrics) and 40-01 (session-report): tests first, lib module, CLI wiring, integration. The planner generated each independently. A planner that could see "you've built 6 CLI subcommands before, here's the task pattern that worked" would produce better plans faster.

The same gap exists at every planning scale:
- **Task level:** Individual task prompts vary in quality. Some lead to clean execution (0 corrections), others generate rework (3+ corrections). There's no feedback loop from execution quality back to planning quality.
- **Phase level:** When `/gsd:plan-phase` runs, it doesn't know that 3 past phases had similar goals and similar task breakdowns.
- **Milestone level:** When `/gsd:new-milestone` creates a roadmap, it doesn't leverage the structural patterns from past milestones to propose phase breakdowns.

## What to Build

### 1. Phase Reuse / Template Library

Detect when a new phase structurally resembles a past phase and suggest reusing the plan skeleton.

**How it works:**
- Build an index of completed plans: extract task types, file patterns, requirement patterns from PLAN.md frontmatter and task tags
- When `/gsd:plan-phase` runs, search the index for plans with similar file patterns, requirement shapes, or goal keywords
- Surface matches: "Phase 42-01 (skill-scorer) is 85% similar to 41-01 (skill-metrics) â€” reuse skeleton?"
- If accepted, pre-populate the plan draft with the matched skeleton, adjusted for the new phase's files and requirements

**Index structure:**
```json
{
  "plan_id": "42-01",
  "phase_goal": "skill relevance scoring",
  "task_pattern": ["test-first", "lib-module", "cli-wiring", "integration-update"],
  "file_patterns": ["get-shit-done/bin/lib/*.cjs", "tests/*.test.cjs", "get-shit-done/bin/gsd-tools.cjs"],
  "requirement_count": 4,
  "correction_rate": 0.12,
  "tags": ["cli-subcommand", "jsonl-persistence", "tdd"]
}
```

**What exists already:**
- PLAN.md files with structured frontmatter (phase, plan, requirements, files_modified)
- SUMMARY.md files with execution outcomes
- phase-benchmarks.jsonl (v9.0) with correction counts per plan

**What's needed:**
- Plan indexer: scan all completed plans, extract structural features, build searchable index
- Similarity scorer: compare new phase goal/files against index entries
- Integration into plan-phase workflow: surface matches before research/planning begins
- Skeleton adapter: take a matched plan and adjust file paths, requirement IDs, names for the new phase

**Effort:** M

### 2. Phase Composition Assistant

When `/gsd:plan-phase` runs, analyze past similar phases and surface their task structures as starting suggestions.

**How this differs from template library:** Template library matches whole plans. The composition assistant matches individual task patterns and composes them. If your new phase needs "TDD setup + lib module + CLI wiring + dashboard integration," it finds the best example of each task type from history and composes a task sequence.

**How it works:**
- Classify each historical task by type: `test-setup`, `lib-module`, `cli-wiring`, `hook-integration`, `workflow-update`, `dashboard-page`, `config-extension`, `documentation`
- When the planner analyzes a new phase's requirements, identify which task types are needed
- For each task type, find the best historical example (lowest correction rate from benchmarks)
- Surface a composed suggestion: "Suggested task sequence based on historical performance: 1. test-setup (from 42-01, 0 corrections), 2. lib-module (from 41-01, 0 corrections), 3. cli-wiring (from 40-01, 1 correction)"

**What's needed:**
- Task type classifier (parse `<task>` tags from historical plans, classify by action keywords and file patterns)
- Per-task-type performance ranking (join with phase-benchmarks.jsonl correction data)
- Composition engine: given required task types, select best examples and arrange into sequence
- Integration into plan-phase research step: surface composition as pre-planning context

**Effort:** M

### 3. Automated Milestone Decomposition

Given a milestone goal, propose a phase breakdown using historical phase structure patterns.

**How it works:**
- Analyze all past milestones: extract phase count, phase goals, dependency chains, effort distribution
- Identify structural patterns: "CLI feature milestones typically have 3-4 phases: infrastructure â†’ core logic â†’ integration â†’ polish"
- When `/gsd:new-milestone` runs and the user describes the goal, match against patterns and propose a draft roadmap
- The draft is a starting point, not a final answer â€” the user refines during the questioning step

**What exists already:**
- ROADMAP.md files for all completed milestones (v1.0-v9.0) with phase goals, dependency chains, plan counts
- `/gsd:new-milestone` questioning step that could consume a draft proposal
- MILESTONES.md with milestone-level stats

**What's needed:**
- Milestone structure analyzer: parse historical roadmaps, extract phase patterns and dependency shapes
- Pattern matcher: given a goal description, find similar past milestones by keyword and scope
- Draft proposal generator: produce a phase breakdown with goals, estimated plan counts, and dependency chain
- Integration into new-milestone workflow: surface proposal during questioning as a starting point

**Effort:** L

### 4. Prompt Quality Scoring

After each plan executes, score the plan's task prompts against correction frequency and surface in `/gsd:digest`.

**How it works:**
- At plan completion, count corrections per task (corrections.jsonl entries filtered by phase+plan+task)
- Compute prompt quality score: `1.0 - (corrections / max_expected_corrections)` where max_expected is calibrated from historical data
- Flag plans with high correction rates: "Plan 42-01 Task 3 had 4 corrections â€” action text may be ambiguous"
- Surface in `/gsd:digest`: "Prompt quality: 87% (6/8 plans had 0-1 corrections, 2 plans had 3+ corrections)"
- Over time, identify which prompt patterns correlate with clean execution vs rework

**What exists already:**
- corrections.jsonl with phase/plan context (v6.0)
- phase-benchmarks.jsonl with correction_count per plan (v9.0)
- `/gsd:digest` already surfaces correction analytics

**What's needed:**
- Per-task correction attribution (extend phase-benchmarks or compute at digest time)
- Prompt quality score formula (calibrated against historical correction distribution)
- Prompt pattern analysis: extract common phrases from high-quality vs low-quality prompts
- `/gsd:digest` section: prompt quality trends
- Optional: feed high-quality prompt patterns back to the planner as examples

**Effort:** M

## Estimated Scope

4 phases, 5-8 plans. Template library and composition assistant are the core; milestone decomposition is the ambitious stretch; prompt scoring is the analytics layer.

## Key Decision Points

1. **Plan index: build-time or query-time?** A pre-built index (JSON file refreshed at milestone completion) is fast to query but stale between milestones. Query-time scanning is always fresh but slower. Recommendation: pre-built index refreshed by `/gsd:complete-milestone`, with a `--rebuild-index` flag for manual refresh.

2. **Composition assistant: suggest or auto-populate?** Suggesting task sequences lets the planner adapt. Auto-populating plan drafts risks the planner treating the draft as final and not adapting. Recommendation: suggest as context input to the planner, not as a pre-filled plan template.

3. **Milestone decomposition: how opinionated?** A generic "3-4 phases" suggestion isn't useful. A specific "Phase 1: infrastructure (like v9.0 Phase 37), Phase 2: core logic (like v9.0 Phases 38-39)" is useful but might anchor the planner too strongly. Recommendation: specific with explicit "adapt as needed" framing.

4. **Prompt scoring calibration:** What correction count is "normal" vs "high"? This needs a baseline from historical data. Recommendation: compute median corrections per task across all milestones, flag tasks >2x median.
